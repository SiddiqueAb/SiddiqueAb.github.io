দ্রুততর হয় SDLC মানলে
💡 SDLC কেন দরকার? যে কেউ চাইলে কোড লিখতে পারে। কিন্তু একটি সফটওয়্যার সময়মতো, বাজেটের মধ্যে, এবং ব্যবহারকারীর চাহিদা পূরণ করে তৈরি করতে হলে একটি কাঠামো দরকার। SDLC সেই কাঠামো। এটি ছাড়া সফটওয়্যার প্রজেক্ট অনেকটা নকশা ছাড়া বাড়ি বানানোর মতো — শুরুতে ভালো মনে হলেও, শেষে সব ভেঙে পড়ে।

SDLC ছাড়া কী হয়?

  • কোড লেখার পর বোঝা যায় ভুল জিনিস বানানো হয়েছে
  • সময় ও বাজেট দ্বিগুণ হয়ে যায়
  • টিমের সদস্যরা একে অপরের কাজ বোঝে না
  • Bug ধরা পড়ে production-এ গিয়ে
  • Client বা user অসন্তুষ্ট হন
  • Maintenance করা অসম্ভব হয়ে পড়ে

SDLC মানলে কী পাবেন?

  • স্পষ্ট পরিকল্পনা — কে কী করবে জানে
  • সময়মতো ডেলিভারি সম্ভব হয়
  • কম Bug, বেশি মান
  • Client বা user সন্তুষ্ট থাকেন
  • ভবিষ্যতে নতুন ফিচার যোগ সহজ হয়
  • টিম কাজ করতে পারে কার্যকরভাবে

📋 Phase 1 · Planning

পরিকল্পনা পর্যায়

সফটওয়্যার তৈরির সবচেয়ে গুরুত্বপূর্ণ ধাপ। এখানে ঠিক হয় — কী বানাবেন, কেন বানাবেন, কতটুকু সময় লাগবে, কতটাকা লাগবে এবং কতটুকু ঝুঁকি আছে।

পরিকল্পনায় কী কী থাকে?

সমস্যা সংজ্ঞায়ন (Problem Definition)
কী সমস্যা সমাধান করতে চাইছেন? কে ভোগান্তিতে পড়ছে? সমস্যাটা কতটা বড়?
সম্ভাব্যতা বিশ্লেষণ (Feasibility Study)
এটা কি করা সম্ভব? — প্রযুক্তিগত, আর্থিক, সময়গত এবং আইনগত দিক থেকে।
রিসোর্স পরিকল্পনা (Resource Planning)
কতজন মানুষ লাগবে? কোন টুলস? কতটা বাজেট? কতদিন?
ঝুঁকি বিশ্লেষণ (Risk Analysis)
কোন কোন সমস্যা হতে পারে? হলে কী করবেন?
প্রজেক্ট চার্টার তৈরি
সব সিদ্ধান্ত একটি দলিলে লেখা হয়। এটি হয় প্রজেক্টের সংবিধান।
🎯 পরিকল্পনার লক্ষ্য "আমরা কি সঠিক জিনিস বানাচ্ছি?" — এই প্রশ্নের উত্তর এখানে ঠিক হয়। পরিকল্পনা ছাড়া কোড লেখা শুরু করলে পরে সব ভেঙে দিতে হবে।

📊 Feasibility Study — ৫ ধরন

  • Technical: প্রযুক্তিগতভাবে সম্ভব?
  • Economic: আর্থিকভাবে লাভজনক?
  • Legal: আইনের বিপরীতে না?
  • Operational: ব্যবহারকারীরা ব্যবহার করবে?
  • Schedule: সময়মতো শেষ করা যাবে?

Solo Developer বনাম Team — পরিকল্পনা পর্যায়

👤 Solo

একা যখন কাজ করবেন

📓Notion বা নোটবুক — আইডিয়া, ফিচার লিস্ট, সময়সূচি এখানে লিখুন। সব মাথায় রাখবেন না।
🗓️Weekly Goal সেট করুন — এই সপ্তাহে কী শেষ হবে তা রবিবার রাতে ঠিক করুন।
⏱️PERT ফর্মুলা ব্যবহার করুন। নিজের অনুমানকে ১.৫ দিয়ে গুণ করুন।
🎯MVP আগে — ৩টির বেশি মূল ফিচার নিয়ে শুরু করবেন না।
💰নিজেই Stakeholder — আপনিই ক্লায়েন্ট, আপনিই ডেভেলপার। নিজেকে প্রশ্ন করুন: "এটা কি আসলে দরকার?"
👥 Team

টিমে যখন কাজ করবেন

📋Kickoff Meeting — সবাই একসাথে বসুন। প্রজেক্টের লক্ষ্য, ভূমিকা, সময়সীমা নির্ধারণ করুন।
📄Project Charter Document তৈরি করুন। সবার সই নিন। এটি হবে প্রজেক্টের চুক্তি।
👔PM বা Tech Lead নির্ধারণ করুন — কে সিদ্ধান্ত নেবে, কে রিপোর্ট করবে।
🗂️Jira / Trello / Linear — প্রজেক্ট ম্যানেজমেন্ট টুল সেটআপ করুন।
📅Sprint Planning — ২ সপ্তাহের Sprints ঠিক করুন। প্রতিটি Sprint-এর Goal নির্দিষ্ট রাখুন।
⚠️ সবচেয়ে বড় ভুল পরিকল্পনা ছাড়াই কোড লেখা শুরু করা। মনে হয় "পরিকল্পনায় সময় নষ্ট" — কিন্তু ভুল পথে দ্রুত গেলে ক্ষতি বেশি। গবেষণা বলে: পরিকল্পনায় ১ ঘণ্টা = development-এ ৪ ঘণ্টা বাঁচায়।

📝 Phase 2 · Requirements

রিকোয়ারমেন্ট বিশ্লেষণ

ব্যবহারকারীরা ঠিক কী চান? সিস্টেমকে কী কী করতে হবে? সব সংগ্রহ করে দলিলবদ্ধ করার ধাপ। এটি ভুল হলে পুরো প্রজেক্ট ভুল দিকে যায়।

🧑‍🤝‍🧑

Functional Requirements

সিস্টেম কী করবে? — লগইন করতে পারবে, অর্ডার দিতে পারবে, রিপোর্ট দেখতে পারবে। এগুলো হলো কার্যকরী চাহিদা।

-- উদাহরণ:
✓ User লগইন করতে পারবে
✓ Product যোগ/বাদ দেওয়া যাবে
✓ Payment প্রক্রিয়া করা যাবে
✓ Invoice তৈরি হবে স্বয়ংক্রিয়ভাবে
⚡

Non-Functional Requirements

সিস্টেম কেমনভাবে করবে? — কত দ্রুত লোড হবে, কতজন একসাথে ব্যবহার করতে পারবে, নিরাপত্তা কেমন হবে।

-- উদাহরণ:
✓ পেজ ২ সেকেন্ডে লোড
✓ ১০,০০০ user একসাথে
✓ ৯৯.৯% uptime
✓ SSL encrypted
🚫

Constraints (সীমাবদ্ধতা)

কী করা যাবে না বা কী মেনে চলতে হবে — বাজেট সীমা, সময়সীমা, নির্দিষ্ট প্রযুক্তি ব্যবহার করতে হবে।

-- উদাহরণ:
✓ বাজেট: ৫ লক্ষ টাকা
✓ সময়: ৩ মাস
✓ শুধু React ব্যবহার করতে হবে
✓ GDPR মানতে হবে

User Story — ব্যবহারকারীর গল্পে লেখুন রিকোয়ারমেন্ট

📖 User Story ফরম্যাট "একজন [ব্যবহারকারী হিসেবে], আমি চাই [কী করতে], যাতে [কেন / কী সুবিধা পাই]।"

উদাহরণ: "একজন গ্রাহক হিসেবে, আমি চাই আমার অর্ডারের স্ট্যাটাস ট্র্যাক করতে, যাতে ডেলিভারির সময় জানতে পারি।"

👤 Solo Developer — রিকোয়ারমেন্ট সংগ্রহ

  1. নিজেই ব্যবহারকারী হন — আপনি কী সমস্যায় পড়েন? সেটাই সমাধান করুন।
  2. ৫ জনকে জিজ্ঞাসা করুন — সম্ভাব্য ব্যবহারকারীদের সাথে ১৫ মিনিট কথা বলুন।
  3. Competitor বিশ্লেষণ — একই ধরনের অ্যাপ কী করছে? কী মিস করছে?
  4. Notion-এ লিস্ট করুন — Feature list: Must have, Nice to have, Future-এ করব।

👥 Team — রিকোয়ারমেন্ট সংগ্রহ

  1. Stakeholder Interview — ক্লায়েন্ট, শেষ ব্যবহারকারী সবার সাথে মিটিং।
  2. Workshop / Brainstorming Session — সবাই মিলে সাদা বোর্ডে লেখুন।
  3. SRS Document তৈরি — Software Requirements Specification। এটা হবে চুক্তির ভিত্তি।
  4. Review ও Sign-off — ক্লায়েন্ট ও টিম সবাই অনুমোদন দিলে তবেই সামনে যান।
🚨 Requirements Creep থেকে সাবধান! প্রজেক্ট শুরুর পর নতুন ফিচার চাওয়া হয় — এটাকে Scope Creep বলে। Solo হলে নিজেই বলুন "না"। Team হলে Change Request Process রাখুন — নতুন ফিচারের জন্য আলাদা estimation করুন এবং ক্লায়েন্টকে বুঝিয়ে বলুন এর cost।

🎨 Phase 3 · Design

সিস্টেম ডিজাইন পর্যায়

রিকোয়ারমেন্ট জানা হলো — এখন পরিকল্পনা করতে হবে সিস্টেম কেমন হবে তার। Architecture, Database, UI — সবকিছুর নকশা এখানে হয়।

🏗️ System Architecture Design

Client (Browser/App)
    ↓ HTTP Request
API Gateway / Load Balancer
    ↓
Backend Server (Node/Django/etc)
    ↓           ↓
Database    Cache (Redis)
(PostgreSQL)  (দ্রুত access)
    ↓
External Services
(Payment, Email, SMS, Storage)

🗄️ Database Design মূলনীতি

  1. Entity চিহ্নিত করুন — কোন কোন "বস্তু" আছে? User, Product, Order, Payment?
  2. Relation নির্ধারণ — One-to-One, One-to-Many, Many-to-Many?
  3. Normalization — ডেটা যাতে duplicate না হয়।
  4. ER Diagram আঁকুন — dbdiagram.io বা Lucidchart দিয়ে।

🔬 Phase 5 · Testing

টেস্টিং ও মান নিয়ন্ত্রণ

সফটওয়্যার কি ঠিকমতো কাজ করছে? ব্যবহারকারী কি সমস্যায় পড়বেন? শিপিং করার আগে সব পরীক্ষা করতে হবে এখানে।

"Testing করলে bugs পাওয়া যায়। Testing না করলে bugs তৈরি হয় না — বরং hidden থাকে এবং production-এ ধরা পড়ে।"

— Software Engineering নীতি

Testing-এর ধরনসমূহ

Testing ধরন কী পরীক্ষা করে কে করে উদাহরণ
Unit Testing একটি function বা component একা কাজ করছে কিনা Developer login() ফাংশন সঠিক token দিচ্ছে কিনা
Integration Testing দুটো module একসাথে কাজ করছে কিনা Developer/QA Login API সঠিক ডেটা ডেটাবেসে save করছে কিনা
System Testing পুরো সিস্টেম একসাথে কাজ করছে কিনা QA Team User শুরু থেকে শেষ পর্যন্ত কাজ করতে পারছে কিনা
User Acceptance Testing (UAT) Client বা User সন্তুষ্ট কিনা Client/End User Client নিজে ব্যবহার করে দেখেন
Performance Testing চাপে সিস্টেম টিকবে কিনা QA/DevOps ১০,০০০ user একসাথে ব্যবহার করলে ভাঙবে না
Security Testing নিরাপত্তা ফাঁকফোকর আছে কিনা Security Expert SQL Injection, XSS attack থেকে সুরক্ষিত কিনা

📋Test Checklist তৈরি করুন — প্রতিটি feature কি কাজ করছে? তালিকা করুন।
🔧Postman — API গুলো test করুন। সব endpoint ঠিকমতো কাজ করছে কিনা দেখুন।
📱বিভিন্ন ডিভাইসে দেখুন — Mobile, Tablet, Desktop। বিভিন্ন browser-এ।
🤝Beta Testers খুঁজুন — ৫-১০ জন বন্ধু বা পরিচিতকে ব্যবহার করতে দিন।
👥 Team

টিমে Testing করুন

👤আলাদা QA Engineer রাখুন — Developer নিজের code ভালোভাবে test করতে পারেন না।

📊Test Coverage রিপোর্ট — কতটুকু code test করা হয়েছে তা মাপুন।
✅Definition of Done — কোন feature কখন "শেষ" হবে তার সংজ্ঞা আগেই ঠিক করুন।
🤖 Automated Testing — কেন শিখবেন? প্রতিবার নতুন ফিচার যোগ করলে কি পুরোনো সব test ম্যানুয়ালি করবেন? সময় কত লাগবে? Automated test লিখলে একটি command চালালেই সব পরীক্ষা হয়ে যায়। Solo developer-ও Automated testing করা উচিত — এটা দীর্ঘমেয়াদে অনেক সময় বাঁচায়।

🌐 Production

সত্যিকারের ব্যবহারকারীরা ব্যবহার করেন। সবচেয়ে সতর্কতা এখানে।

⚙️ CI/CD — স্বয়ংক্রিয় Deployment

CI/CD কী? Continuous Integration / Continuous Deployment। কোড push করলেই স্বয়ংক্রিয়ভাবে test চলে এবং deploy হয়ে যায়। মানুষকে ম্যানুয়ালি কিছু করতে হয় না।
# GitHub Actions উদাহরণ
name: Deploy to Production
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - npm test   # আগে test
      - npm build  # build করো
      - deploy     # তারপর deploy

📦 Deployment Checklist

  • সব test pass হয়েছে
  • Environment variables সেট আছে
  • Database backup নেওয়া হয়েছে
  • SSL certificate আছে
  • Domain কানেক্ট আছে
  • Error monitoring সেট আপ (Sentry)
  • Rollback plan আছে
  • Team informed করা হয়েছে

🔧 Phase 7 · Maintenance

রক্ষণাবেক্ষণ ও আপডেট

লঞ্চের পরও কাজ শেষ হয় না। বরং এখানে থেকেই আসল দীর্ঘমেয়াদী কাজ শুরু। ব্যবহারকারীর feedback নিন, bug fix করুন, নতুন ফিচার যোগ করুন।

🐛

Corrective Maintenance

Bug fix করা। ব্যবহারকারীরা সমস্যা রিপোর্ট করলে সেটা ঠিক করা।

Priority: Critical bug → ২৪ঘণ্টা | Major → ৭ দিন | Minor → পরের sprint
⬆️

Adaptive Maintenance

পরিবেশ পরিবর্তনের সাথে মানিয়ে নেওয়া। নতুন OS, browser, API version সাপোর্ট।

উদাহরণ: iOS আপডেটে অ্যাপ ভেঙে গেলে fix করা
✨

Perfective Maintenance

নতুন ফিচার যোগ করা, performance উন্নত করা, ব্যবহারযোগ্যতা বাড়ানো।

চলমান: User feedback থেকে নতুন ফিচার আইডিয়া নিন

📊 Monitoring — সবসময় নজর রাখুন

  • Uptime Monitor: Sitecheck, UptimeRobot — সাইট ডাউন হলে alert পাবেন
  • Error Tracking: Sentry — কোন error হচ্ছে, কোথায়, কতবার
  • Analytics: Google Analytics / Plausible — ব্যবহারকারী কী করছেন
  • Performance: Lighthouse — সাইট কত দ্রুত
  • Log Management: যা ঘটছে তার রেকর্ড রাখুন

SDLC Models

কোন মডেল কখন ব্যবহার করবেন?

SDLC-এর ধাপগুলো একই — কিন্তু সেগুলো কীভাবে সাজাবেন তার বিভিন্ন পদ্ধতি আছে। প্রজেক্টের ধরন বুঝে সঠিক মডেল বেছে নিন।

🌊 Waterfall Model

একটি ধাপ শেষ না হলে পরেরটিতে যাওয়া যায় না। পানির মতো একদিকে প্রবাহিত হয়।

📋 পরিকল্পনা (সম্পূর্ণ)
   ↓
📝 রিকোয়ারমেন্ট (সম্পূর্ণ)
   ↓
🎨 ডিজাইন (সম্পূর্ণ)
   ↓
💻 ডেভেলপমেন্ট (সম্পূর্ণ)
   ↓
🔬 টেস্টিং (সম্পূর্ণ)
   ↓
🚀 ডিপ্লয়মেন্ট

✅ কখন ব্যবহার করবেন

  • রিকোয়ারমেন্ট সম্পূর্ণ স্পষ্ট ও পরিবর্তন হবে না
  • সরকারি বা প্রতিরক্ষা প্রজেক্ট
  • নির্মাণ বা হার্ডওয়্যার প্রজেক্ট
  • Documentation বেশি দরকার

❌ সমস্যা

  • পরিবর্তনে অনেক পিছিয়ে যেতে হয়
  • শেষে গিয়ে বড় সমস্যা ধরা পড়ে
  • Client শেষে দেখেন — তখন অনেক দেরি

🔄 Agile Methodology

ছোট ছোট চক্রে (Iteration/Sprint) কাজ করা। প্রতিটি চক্রে একটি কার্যকরী পণ্য তৈরি হয়। পরিবর্তনকে স্বাগত জানায়।

Agile Manifesto-এর মূলনীতি ১. Process-এর চেয়ে মানুষ ও যোগাযোগ গুরুত্বপূর্ণ
২. Documentation-এর চেয়ে কার্যকরী সফটওয়্যার গুরুত্বপূর্ণ
৩. Contract-এর চেয়ে Customer সহযোগিতা গুরুত্বপূর্ণ
৪. পরিকল্পনা অনুসরণের চেয়ে পরিবর্তনে সাড়া দেওয়া গুরুত্বপূর্ণ

✅ কখন ব্যবহার করবেন

  • রিকোয়ারমেন্ট পরিবর্তন হতে পারে
  • Startup ও product কোম্পানি
  • Client নিয়মিত feedback দেবেন
  • দ্রুত market-এ যেতে চান

🎯 Solo Agile

Solo developer-ও Agile করতে পারেন। প্রতি সপ্তাহ = একটি Sprint। Backlog তৈরি করুন। Weekly Review করুন।

🏉 Scrum Framework

Agile-এর সবচেয়ে জনপ্রিয় implementation। নির্দিষ্ট ভূমিকা, ceremonies এবং artifacts দিয়ে কাজ করে।

ভূমিকাদায়িত্ব
Product Ownerকী বানাবেন ঠিক করেন। Backlog manage করেন।
Scrum Masterপ্রক্রিয়া নিশ্চিত করেন। বাধা দূর করেন।
Dev Teamডিজাইন, কোড, test করেন।

📅 Scrum Ceremonies

Sprint Planning (২-৪ ঘণ্টা)
Sprint-এ কী কী কাজ নেওয়া হবে তা ঠিক করা।
Daily Standup (১৫ মিনিট)
প্রতিদিন: কাল কী করলাম, আজ কী করব, কোথায় সমস্যা।
Sprint Review (১-২ ঘণ্টা)
Stakeholder-দের কাছে Sprint-এর ফলাফল দেখানো।
Retrospective (১ ঘণ্টা)
কী ভালো হলো, কী উন্নত করব পরের Sprint-এ।

📌 Kanban System

Sprint নেই, নির্দিষ্ট সময়সীমা নেই। কাজ আসে, হয়, পরবর্তী কাজে যায়। সহজ এবং flexible। Solo developer-এর জন্য আদর্শ।

Solo Developer-এর জন্য Kanban Trello বা Notion board তৈরি করুন। কলাম: Backlog → To Do → Doing → Review → Done। একসাথে ৩-এর বেশি কাজ "Doing"-এ রাখবেন না।

🌀 Spiral Model

Risk-driven পদ্ধতি। প্রতিটি চক্রে risk বিশ্লেষণ করা হয়। বড় ও জটিল প্রজেক্টের জন্য।

চক্র ১: Prototype তৈরি
  → Risk বিশ্লেষণ
  → Client review

চক্র ২: Basic features
  → Risk বিশ্লেষণ
  → Client review

চক্র ৩: Full features
  → Risk বিশ্লেষণ
  → Final product

✅ কখন ব্যবহার করবেন

  • বড় ও complex প্রজেক্ট
  • High risk প্রজেক্ট
  • Medical বা Defense software
  • দীর্ঘমেয়াদী প্রজেক্ট
Solo-র জন্য? সাধারণত Solo developer Spiral ব্যবহার করেন না। এটি বড় টিম ও বড় বাজেটের জন্য।

🚀 Lean Startup / MVP Approach

Solo developer এবং Startup-এর জন্য সেরা। দ্রুত Minimum Viable Product বানান। Real user থেকে feedback নিন। শিখুন ও আবার বানান।

Build → Measure → Learn সবচেয়ে ছোট কার্যকরী পণ্য বানান → আসল ব্যবহারকারীদের কাছে দিন → তারা কী করছে মাপুন → শিখুন → উন্নত করুন → আবার বানান।
আইডিয়া → MVP (২ সপ্তাহ)
  ↓
১০ জন real user → Feedback
  ↓
Pivot বা Improve → v2.0
  ↓
১০০ জন user → আরো Feedback
  ↓
Scale up করুন

MVP-তে কী রাখবেন?

  • Must Have: মূল সমস্যা সমাধান করে এমন ফিচার
  • Should Have: পরের version-এ
  • Nice to Have: এখন না

⚡ Solo-র জন্য Lean Timeline

সপ্তাহ ১: পরিকল্পনা ও ডিজাইন → সপ্তাহ ২-৩: MVP তৈরি → সপ্তাহ ৪: Testing ও Feedback → সপ্তাহ ৫+: Iterate

👤 Solo Developer হিসেবে

  • সব কিছু নিজে করার চেষ্টা করুন — Full Stack শিখুন
  • তবে সবকিছু scratch থেকে বানাবেন না। Library ব্যবহার করুন
  • নিয়মিত ছোট ছোট commit করুন — ইতিহাস রাখুন
  • শেষ করুন একটা জিনিস, তারপর পরের কাজ
  • Documentation লিখুন — ৬ মাস পরে নিজেই ভুলে যাবেন
  • অন্য developer-দের সাথে community-তে যোগ দিন

👥 Team Member হিসেবে

  • আগে শুনুন, তারপর কথা বলুন
  • নিজের সমস্যা সময়মতো জানান — লুকাবেন না
  • Code review-কে সমালোচনা মনে করবেন না — শেখার সুযোগ
  • সবার কাজকে সম্মান করুন — কেউ কম গুরুত্বপূর্ণ নয়
  • নিজের কাজ estimate করতে শিখুন — সৎভাবে বলুন
  • টিমের সাফল্য আপনার সাফল্য — ego বাদ দিন
