Numeric ডেটা টাইপ (সংখ্যা)

ℹ️
কখন Numeric ব্যবহার করবেন?

যেকোনো সংখ্যা — বয়স, পরিমাণ, মূল্য, গণনা, আইডি — এই ধরনের ডেটার জন্য Numeric টাইপ ব্যবহার করুন।

ডেটা টাইপ স্টোরেজ রেঞ্জ (Signed) রেঞ্জ (Unsigned) ব্যবহার
TINYINT 1 Byte -128 থেকে 127 0 থেকে 255 বয়স, রেটিং (1-5), ছোট কাউন্টার
SMALLINT 2 Bytes -32,768 থেকে 32,767 0 থেকে 65,535 বছর, পোস্টাল কোড, স্টক পরিমাণ
MEDIUMINT 3 Bytes -8,388,608 থেকে 8,388,607 0 থেকে 16,777,215 মাঝারি কাউন্টার, জনসংখ্যা
INT 4 Bytes -2.1B থেকে 2.1B 0 থেকে 4.29B প্রাইমারি কী, অর্ডার আইডি, সাধারণ সংখ্যা
BIGINT 8 Bytes ±9.2 কোয়াড্রিলিয়ন 0 থেকে 18.4 কোয়াড্রিলিয়ন Twitter ID, বড় ট্রানজেকশন, সোশ্যাল মিডিয়া
FLOAT 4 Bytes ±3.4×10³⁸ (7 দশমিক) — বৈজ্ঞানিক পরিমাপ, গড়, তাপমাত্রা
DOUBLE 8 Bytes ±1.7×10³⁰⁸ (15 দশমিক) — উচ্চ নির্ভুলতার গণনা, জিওগ্রাফিক কোঅর্ডিনেট
DECIMAL(p,s) প্রতি 9 digit = 4 Bytes নির্ভুল দশমিক (65 ডিজিট পর্যন্ত) — টাকা, মূল্য, ট্যাক্স — কখনো ভুল হওয়া যাবে না
BOOLEAN 1 Byte 0 বা 1 — সক্রিয়/নিষ্ক্রিয়, হ্যাঁ/না, পাস/ফেইল
TINYINT
1 Byte
সবচেয়ে ছোট পূর্ণসংখ্যা। যেখানে মান সীমিত ও ছোট।
✔ ব্যবহার করুন: বয়স, স্টার রেটিং, সক্রিয় স্ট্যাটাস, মাস নম্বর
UNSIGNED: 0–255 | SIGNED: -128–127
INT
4 Bytes
সবচেয়ে পরিচিত ও ব্যবহৃত সংখ্যা টাইপ। PRIMARY KEY এর ডিফল্ট পছন্দ।
✔ ব্যবহার করুন: ID, পণ্যের পরিমাণ, ভোটের সংখ্যা, অর্ডার নম্বর
UNSIGNED: 0–4,294,967,295
DECIMAL(10,2)
নির্ভুল
100% নির্ভুল দশমিক সংখ্যা। আর্থিক লেনদেনের জন্য অপরিহার্য।
✔ ব্যবহার করুন: মূল্য, বেতন, ট্যাক্স, ব্যালেন্স — টাকার হিসাব সবসময় DECIMAL
DECIMAL(10,2) → 99999999.99 পর্যন্ত
⚠️
টাকার হিসাবে FLOAT/DOUBLE ব্যবহার করবেন না!

FLOAT এবং DOUBLE এ rounding error হয়। উদাহরণ: 0.1 + 0.2 = 0.30000000000000004 হতে পারে! সবসময় DECIMAL ব্যবহার করুন।

SQL উদাহরণ CREATE TABLE products ( id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, name VARCHAR(100) NOT NULL, quantity SMALLINT UNSIGNED DEFAULT 0, -- স্টক পরিমাণ price DECIMAL(10,2) NOT NULL, -- টাকার মূল্য ✓ rating TINYINT UNSIGNED, -- 1-5 রেটিং is_active BOOLEAN DEFAULT TRUE, -- হ্যাঁ/না views BIGINT UNSIGNED DEFAULT 0, -- ভিউ কাউন্টার temperature FLOAT -- তাপমাত্রা (নির্ভুলতা লাগে না) );
📝

String ডেটা টাইপ (টেক্সট)

ডেটা টাইপ সর্বোচ্চ আকার স্টোরেজ পদ্ধতি কখন ব্যবহার করবেন
CHAR(n) 255 char n Bytes (স্থির) নির্দিষ্ট দৈর্ঘ্যের ডেটা: দেশ কোড, OTP, লিঙ্গ
VARCHAR(n) 65,535 char দৈর্ঘ্য + 1-2 Bytes পরিবর্তনশীল টেক্সট: নাম, ইমেইল, শিরোনাম
TINYTEXT 255 char দৈর্ঘ্য + 1 Byte ছোট বিবরণ, সংক্ষিপ্ত মন্তব্য
TEXT 65,535 char দৈর্ঘ্য + 2 Bytes ব্লগ পোস্ট, পণ্যের বিবরণ, মন্তব্য
MEDIUMTEXT 16 MB দৈর্ঘ্য + 3 Bytes বড় আর্টিকেল, HTML কন্টেন্ট
LONGTEXT 4 GB দৈর্ঘ্য + 4 Bytes বই, লগ ফাইল, বিশাল ডকুমেন্ট
ENUM 65,535 অপশন 1-2 Bytes নির্দিষ্ট তালিকা: লিঙ্গ, স্ট্যাটাস, বিভাগ
SET 64 সদস্য 1-8 Bytes একাধিক মান নির্বাচন: ট্যাগ, অনুমতি
💡
CHAR বনাম VARCHAR — কোনটা বেছে নেবেন?

CHAR(n) ব্যবহার করুন যখন ডেটার দৈর্ঘ্য সবসময় একই থাকে (যেমন: দেশ কোড 'BD', OTP '123456')। কারণ এটি দ্রুত। VARCHAR(n) ব্যবহার করুন যখন দৈর্ঘ্য পরিবর্তন হয় — এটি স্থান সাশ্রয় করে।

SQL উদাহরণ CREATE TABLE users ( id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, username VARCHAR(50) NOT NULL UNIQUE, -- পরিবর্তনশীল দৈর্ঘ্য email VARCHAR(150) NOT NULL UNIQUE, -- ইমেইল সর্বোচ্চ ~150 phone CHAR(11) NOT NULL, -- সবসময় 11 সংখ্যা country_code CHAR(2) DEFAULT 'BD', -- সবসময় 2 অক্ষর gender ENUM('male','female','other'), -- নির্দিষ্ট তালিকা bio TEXT, -- দীর্ঘ বায়ো full_article MEDIUMTEXT, -- বড় আর্টিকেল status ENUM('active','inactive','banned') DEFAULT 'active' );
🔍
INDEX এ TEXT ব্যবহার সীমিত!

TEXT কলামে সরাসরি INDEX দেওয়া যায় না। পরিবর্তে prefix index ব্যবহার করুন: INDEX (bio(100)) — প্রথম 100 অক্ষর ইনডেক্স করবে।

Date/Time ডেটা টাইপ (তারিখ ও সময়)

ডেটা টাইপ স্টোরেজ ফরম্যাট রেঞ্জ ব্যবহার
DATE 3 Bytes YYYY-MM-DD 1000-01-01 থেকে 9999-12-31 জন্মতারিখ, ছুটির তারিখ, ডেডলাইন
TIME 3 Bytes HH:MM:SS -838:59:59 থেকে 838:59:59 ব্যবসার সময়, শিডিউল, সময়কাল
DATETIME 8 Bytes YYYY-MM-DD HH:MM:SS 1000 থেকে 9999 সাল অর্ডারের সময়, লগইন সময়, সাধারণ টাইমস্ট্যাম্প
TIMESTAMP 4 Bytes YYYY-MM-DD HH:MM:SS 1970-01-01 থেকে 2038-01-19 created_at, updated_at — auto update সুবিধা আছে
YEAR 1 Byte YYYY 1901 থেকে 2155 নির্মাণ বছর, স্নাতক সাল
TIMESTAMP
4 Bytes
Timezone সচেতন। রেকর্ড তৈরি ও আপডেটের সময় স্বয়ংক্রিয়ভাবে সেট হয়।
✔ ব্যবহার করুন: created_at, updated_at, last_login
⚠️ সীমা: 2038 সাল পর্যন্ত (Y2K38 সমস্যা)
DATETIME
8 Bytes
২০৩৮ সালের পরের তারিখের জন্য। Timezone সরাসরি সংরক্ষণ করে না।
✔ ব্যবহার করুন: ভবিষ্যতের অ্যাপয়েন্টমেন্ট, ঐতিহাসিক ডেটা
রেঞ্জ: 1000-01-01 থেকে 9999-12-31
DATE
3 Bytes
শুধু তারিখ, সময় ছাড়া। সময়ের দরকার না থাকলে এটি বেশি সাশ্রয়ী।
✔ ব্যবহার করুন: জন্মতারিখ, অর্থ-বছর, ছুটির তালিকা
DATETIME এর চেয়ে 5 Bytes সাশ্রয়!
SQL উদাহরণ CREATE TABLE orders ( id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, user_id INT UNSIGNED NOT NULL, birth_date DATE, -- শুধু তারিখ, 3 bytes delivery_at DATETIME, -- নির্দিষ্ট ডেলিভারি সময় grad_year YEAR, -- শুধু বছর, 1 byte created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, -- স্বয়ং সেট updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP -- স্বয়ং আপডেট );

অন্যান্য ডেটা টাইপ

ডেটা টাইপ স্টোরেজ বর্ণনা ব্যবহার
TINYBLOB 255 Bytes + 1 ছোট বাইনারি ডেটা ছোট ছবি থাম্বনেইল (সুপারিশ করা হয় না)
BLOB 64 KB + 2 মাঝারি বাইনারি ডকুমেন্ট, অডিও ক্লিপ
MEDIUMBLOB 16 MB + 3 বড় বাইনারি ছবি, পিডিএফ
LONGBLOB 4 GB + 4 বিশাল বাইনারি ভিডিও (ফাইল সিস্টেমই ভালো)
JSON পরিবর্তনশীল JSON ফরম্যাট নমনীয় স্কিমা, মেটাডেটা, কনফিগ
UUID / CHAR(36) 36 Bytes ইউনিক আইডি ডিস্ট্রিবিউটেড সিস্টেম, পাবলিক API
📁
ডাটাবেজে ছবি/ফাইল সংরক্ষণ করবেন না!

BLOB এ ছবি/ভিডিও রাখলে ডাটাবেজ ধীর হয়ে যায় এবং ব্যাকআপ বিশাল হয়। পরিবর্তে ফাইল ফোল্ডার বা AWS S3 তে রাখুন এবং ডাটাবেজে শুধু পাথ/URL সংরক্ষণ করুন।

JSON ডেটা টাইপ উদাহরণ CREATE TABLE user_settings ( id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, user_id INT UNSIGNED NOT NULL, prefs JSON, -- নমনীয় সেটিংস avatar VARCHAR(255) -- ছবির URL (BLOB নয়!) ✓ ); -- JSON কলামে ডেটা ইনসার্ট INSERT INTO user_settings (user_id, prefs) VALUES ( 1, '{"theme":"dark","lang":"bn","notifications":true}' ); -- JSON থেকে নির্দিষ্ট মান বের করুন SELECT JSON_EXTRACT(prefs, '$.theme') AS theme FROM user_settings;

স্টোরেজ গণনা — কীভাবে হিসাব করবেন

📐
মূল সূত্র:

একটি রো'র মোট আকার = সব কলামের আকারের যোগফল + ওভারহেড (≈ 20-60 Bytes প্রতি রো)

🏪 একটি ই-কমার্স টেবিলের স্টোরেজ উদাহরণ
কলাম ডেটা টাইপ প্রতি রো আকার ব্যাখ্যা
id INT UNSIGNED 4 Bytes সর্বদা 4 Bytes
name VARCHAR(100) দৈর্ঘ্য + 2 Bytes "Samsung TV" = 10+2 = 12 Bytes
price DECIMAL(10,2) 5 Bytes 10 ডিজিটের DECIMAL
quantity SMALLINT UNSIGNED 2 Bytes সর্বদা 2 Bytes
category ENUM (10 অপশন) 1 Byte ≤255 অপশন = 1 Byte
description TEXT দৈর্ঘ্য + 2 Bytes "বাংলাদেশে তৈরি…" = পরিবর্তনশীল
created_at TIMESTAMP 4 Bytes সর্বদা 4 Bytes
মোট (গড়) ~200 Bytes/রো ওভারহেড সহ আনুমানিক
স্টোরেজ হিসাব -- ১০ লক্ষ পণ্য থাকলে: 1,000,000 রো × 200 Bytes = 200,000,000 Bytes = 200 MB (মোটামুটি) -- ১ কোটি রেকর্ড: 10,000,000 × 200 = 2 GB শুধু ডেটা (+ INDEX আলাদা স্থান নেয়) -- MySQL এ টেবিলের আকার দেখুন: SELECT table_name, ROUND((data_length + index_length) / 1024 / 1024, 2) AS 'Size (MB)' FROM information_schema.tables WHERE table_schema = 'আপনার_ডাটাবেজের_নাম' ORDER BY (data_length + index_length) DESC;
📊 VARCHAR এর স্টোরেজ গণনা

VARCHAR এর আকার নির্ভর করে আসল ডেটার উপর, সর্বোচ্চ সীমার উপর নয়।

-- VARCHAR(255) তে 'Rahim' সংরক্ষণ করলে: 'Rahim' = 5 অক্ষর × 1 Byte + 1 Byte (দৈর্ঘ্য) = 6 Bytes -- VARCHAR(255) তে 'মোহাম্মদ রহিম' (UTF-8) সংরক্ষণ করলে: 'মোহাম্মদ রহিম' = 13 অক্ষর × 3 Bytes (UTF-8) + 1 Byte = 40 Bytes -- CHAR(10) তে 'BD' সংরক্ষণ করলে: 'BD' সবসময় = 10 Bytes (বাকি 8 স্পেস দিয়ে পূরণ)

কোন টাইপ কখন বেছে নেবেন?

১

📌 সংখ্যা সংরক্ষণ করতে চান?

টাকা/মূল্য → DECIMAL(10,2) ID/গণনা → INT / BIGINT ছোট সংখ্যা → TINYINT/SMALLINT দশমিক (অনির্ভুল) → FLOAT/DOUBLE
২

📝 টেক্সট সংরক্ষণ করতে চান?

নির্দিষ্ট দৈর্ঘ্য → CHAR(n) পরিবর্তনশীল (≤255) → VARCHAR দীর্ঘ লেখা → TEXT/MEDIUMTEXT নির্দিষ্ট তালিকা → ENUM
৩

📅 তারিখ/সময় সংরক্ষণ করতে চান?

শুধু তারিখ → DATE তারিখ+সময় (2038 পরে) → DATETIME Auto-update দরকার → TIMESTAMP শুধু বছর → YEAR
৪

🔗 বিশেষ ডেটা সংরক্ষণ করতে চান?

নমনীয় স্কিমা → JSON হ্যাঁ/না → BOOLEAN (TINYINT(1)) ডিস্ট্রিবিউটেড ID → CHAR(36) UUID
⚡ CHAR বনাম VARCHAR — তুলনা
বৈশিষ্ট্য CHAR(n) VARCHAR(n)
স্টোরেজ সবসময় n Bytes আসল দৈর্ঘ্য + 1-2 Bytes
গতি ✓ দ্রুততর ~ মাঝারি
স্থান ✗ বেশি নষ্ট ✓ সাশ্রয়ী
উপযুক্ত নির্দিষ্ট দৈর্ঘ্য (OTP, কোড) পরিবর্তনশীল (নাম, ইমেইল)

সম্পূর্ণ উদাহরণ -- ব্যবহারকারী টেবিল CREATE TABLE users ( id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, -- 4B name VARCHAR(100) NOT NULL, -- পরিবর্তনশীল email VARCHAR(150) NOT NULL UNIQUE, -- সর্বোচ্চ ~100 char phone CHAR(11), -- সবসময় 11 সংখ্যা password CHAR(60) NOT NULL, -- bcrypt hash সবসময় 60 dob DATE, -- জন্মতারিখ 3B gender ENUM('male','female','other'), -- 1B balance DECIMAL(12,2) DEFAULT 0.00, -- ওয়ালেট টাকা is_verified BOOLEAN DEFAULT FALSE, -- 1B created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP -- 4B ); -- পণ্য টেবিল CREATE TABLE products ( id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, name VARCHAR(200) NOT NULL, slug VARCHAR(220) NOT NULL UNIQUE, -- URL-friendly price DECIMAL(10,2) NOT NULL, -- কখনো FLOAT নয়! discount TINYINT UNSIGNED DEFAULT 0, -- 0-100% 1B stock SMALLINT UNSIGNED DEFAULT 0, -- স্টক 2B weight DECIMAL(8,3), -- কেজিতে ওজন description TEXT, -- দীর্ঘ বিবরণ meta JSON, -- নমনীয় অ্যাট্রিবিউট views INT UNSIGNED DEFAULT 0, -- ভিউ কাউন্টার 4B created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP ); -- অর্ডার টেবিল CREATE TABLE orders ( id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY, -- বড় কাউন্টার 8B user_id INT UNSIGNED NOT NULL, total DECIMAL(12,2) NOT NULL, status ENUM('pending','paid','shipped','done','cancelled') DEFAULT 'pending', address TEXT NOT NULL, delivered_at DATETIME, -- ভবিষ্যত তারিখ created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );
📱 সোশ্যাল মিডিয়া স্কিমা
CREATE TABLE posts ( id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY, -- Twitter-like বড় ID user_id INT UNSIGNED NOT NULL, content VARCHAR(500) NOT NULL, -- টুইটের মতো media_url VARCHAR(500), -- ছবির URL (BLOB নয়) likes INT UNSIGNED DEFAULT 0, shares INT UNSIGNED DEFAULT 0, is_public BOOLEAN DEFAULT TRUE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); -- স্টোরেজ হিসাব: গড় পোস্ট ~300 Bytes -- ১০ কোটি পোস্ট = 100M × 300 = 30 GB ডেটা -- + INDEX ≈ আরো 10-15 GB -- মোট আনুমানিক: ~45 GB

সেরা অনুশীলন (Best Practices)

১. সবসময় ক্ষুদ্রতম উপযুক্ত টাইপ ব্যবহার করুন
২. টাকার হিসাবে সবসময় DECIMAL, কখনো FLOAT নয়
৩. NULL অনুমতি দেওয়ার আগে ভাবুন — NOT NULL দ্রুততর
৪. VARCHAR(255) ডিফল্ট না করে প্রকৃত প্রয়োজন অনুযায়ী সেট করুন
৫. created_at/updated_at এ সবসময় TIMESTAMP ব্যবহার করুন
৬. BLOB এ ফাইল না রেখে পাথ/URL সংরক্ষণ করুন
