ACID Properties

ACID কি এবং কেন গুরুত্বপূর্ণ?

Database transaction-এর চারটি মূল property হলো ACID। এগুলো ছাড়া কোনো financial বা critical system reliable হতে পারে না।

Propertyমানে কিব্যর্থ হলে কী হয়Real উদাহরণ
Atomicity সব কাজ হবে অথবা কিছুই না Partial update — অর্ধেক টাকা কাটা, credit নেই Fund transfer এর দুটো UPDATE

02

Transaction Control — START, COMMIT, ROLLBACK

Transaction কিভাবে শুরু ও শেষ করতে হয়

MySQL-এ default autocommit = ON — মানে প্রতিটি statement নিজেই একটি transaction। Production code-এ manually transaction control করতে হয়।

SQL — Basic Transaction Structure 
-- Transaction শুরু করো
START TRANSACTION;
-- অথবা: BEGIN;

-- তোমার SQL operations
UPDATE wallets
  SET balance = balance - 500
  WHERE user_id = 101;

UPDATE wallets
  SET balance = balance + 500
  WHERE user_id = 202;

-- সব ঠিক? Save করো
COMMIT;

-- কোনো error? সব বাতিল
ROLLBACK;
🏦 Real Scenario — Wallet Fund Transfer

User A (id=101) থেকে User B (id=202)-তে ৫০০ টাকা transfer। দ্বিতীয় UPDATE fail করলে (user B নেই) — প্রথম debit-ও ROLLBACK হবে। টাকা হারিয়ে যাবে না।

Application Code-এ Transaction Pattern (PHP Example)

PHP — Transaction with Error Handling 
// ✅ সঠিক পদ্ধতি
try {
    $pdo->beginTransaction();

    $pdo->exec("UPDATE wallets SET balance = balance - 500 WHERE user_id = 101");
    $pdo->exec("UPDATE wallets SET balance = balance + 500 WHERE user_id = 202");
    $pdo->exec("INSERT INTO transfer_log (from_id, to_id, amount) VALUES (101, 202, 500)");

    $pdo->commit(); // সব ঠিক, save করো

} catch (Exception $e) {
    $pdo->rollBack(); // কোনো error, সব বাতিল
    throw $e;
}

SAVEPOINT — Partial Rollback

SQL — SAVEPOINT Usage 
START TRANSACTION;

INSERT INTO orders (user_id, amount) VALUES (1, 1000);

SAVEPOINT order_placed;  -- এখানে একটা checkpoint

INSERT INTO payments (order_id, method) VALUES (99, 'card');

-- Payment fail? শুধু payment rollback করো, order রাখো
ROLLBACK TO SAVEPOINT order_placed;

-- Order টা save করো
COMMIT;
  • Transaction open রেখে অন্য API call বা user input wait করা — লম্বা transaction = lock held longer = অন্য query blocked
  • BEGIN করার পর exception catch না করা — transaction কখনো commit/rollback হয় না, connection-এ ঝুলে থাকে
  • MySQL-এ nested BEGIN ব্যবহার — inner BEGIN আসলে outer-কে implicit COMMIT করে দেয়!
03

Isolation Levels

Isolation Level কী এবং কোনটা কখন ব্যবহার করবো?

Isolation level নির্ধারণ করে একটি চলমান transaction অন্য concurrent transaction-এর data কতটুকু দেখতে পাবে। যত বেশি isolation, তত বেশি safety — কিন্তু performance কমে।

Isolation Level Dirty Read Non-Repeatable Read Phantom Read ব্যবহার করো কখন
READ UNCOMMITTED ✓ হয় ✓ হয় ✓ হয় প্রায় কখনো না
READ COMMITTED ✗ হয় না ✓ হয় ✓ হয় Analytics, reporting
REPEATABLE READ ⭐ ✗ হয় না ✗ হয় না ~ InnoDB-তে নেই* Default — বেশিরভাগ app
SERIALIZABLE ✗ হয় না ✗ হয় না ✗ হয় না Financial critical, slow
💡 InnoDB REPEATABLE READ-এর বিশেষত্ব

InnoDB MVCC (Multi-Version Concurrency Control) ব্যবহার করে, তাই Phantom Read সাধারণত হয় না। কিন্তু FOR UPDATE দিলে Gap Lock নেয়, তখন phantom prevent হয়।

SQL — Isolation Level Set করা 
-- শুধু এই session-এর জন্য
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- পরবর্তী transaction-এর জন্য
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Global (সব নতুন connection)
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- Current level দেখো
SELECT @@transaction_isolation;

Dirty Read — কী এবং কেন বিপজ্জনক

SQL — Dirty Read Example (READ UNCOMMITTED) 
-- Connection A (টাকা transfer করছে)
START TRANSACTION;
UPDATE wallets SET balance = balance + 1000 WHERE id = 5;
-- এখনো COMMIT করেনি...

-- Connection B (READ UNCOMMITTED mode-এ)
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT balance FROM wallets WHERE id = 5;
-- ⚠️ uncommitted +1000 দেখাবে! (dirty read)

-- Connection A তখন ROLLBACK করলো
ROLLBACK;
-- B যা দেখেছিল সেটা false data ছিল — বিপদ!
  • Analytics query-তে default REPEATABLE READ রাখা — long-running read transaction locks নিতে পারে। READ COMMITTED ব্যবহার করো।
  • SERIALIZABLE সব জায়গায় ব্যবহার — deadlock rate বাড়ে, throughput কমে
04

SELECT … FOR UPDATE

Pessimistic Locking — আগে lock নাও, তারপর কাজ করো

SELECT ... FOR UPDATE হলো pessimistic locking-এর মূল tool। এটি selected row(s)-এ exclusive lock নেয় — অন্য কোনো transaction এই row modify বা exclusive lock নিতে পারবে না যতক্ষণ না তুমি COMMIT/ROLLBACK করো।

কে কী করতে পারবেFOR UPDATE lock থাকা অবস্থায়
অন্য transaction — plain SELECT✅ পারবে (MVCC snapshot দেখবে)
অন্য transaction — SELECT FOR UPDATE❌ block হবে, wait করবে
অন্য transaction — UPDATE/DELETE❌ block হবে, wait করবে
অন্য transaction — LOCK IN SHARE MODE❌ block হবে
SQL — SELECT FOR UPDATE Syntax 
START TRANSACTION;

-- Row lock নাও
SELECT id, balance
FROM wallets
WHERE user_id = 101
FOR UPDATE;

-- এখন balance পড়া নিরাপদ — অন্য কেউ পরিবর্তন করতে পারবে না
UPDATE wallets
  SET balance = balance - 300
  WHERE user_id = 101;

COMMIT;
🔐 Real Scenario — Wallet Debit (Race Condition থেকে রক্ষা)

User-এর balance ৫০০ টাকা। দুটো payment request একসাথে আসলো — ৪০০ টাকা করে দুটো payment। FOR UPDATE ছাড়া দুজনেই ৫০০ দেখবে, দুজনেই ৪০০ কাটবে — balance -৩০০ হবে!

FOR UPDATE দিলে — দ্বিতীয় transaction BLOCK হবে। প্রথমটা শেষ হলে দ্বিতীয়টা দেখবে balance ১০০, ৪০০ কাটতে পারবে না → ROLLBACK।

Step-by-Step Execution

  • 1
    Connection A: START TRANSACTION → SELECT balance FROM wallets WHERE user_id=101 FOR UPDATE → balance = ৫০০, row locked ✅
  • 2
    Connection B: START TRANSACTION → SELECT balance FROM wallets WHERE user_id=101 FOR UPDATE → BLOCK হয়ে wait করে
  • 3
    Connection A: UPDATE wallets SET balance = balance - 400 WHERE user_id=101 → COMMIT → balance = ১০০, lock release
  • 4
    Connection B: এখন lock পায় → balance = ১০০ দেখে → ৪০০ কাটতে পারে না → application-level check → ROLLBACK
SQL — Complete Safe Debit Pattern 
START TRANSACTION;

-- ১. Lock নিয়ে balance পড়ো
SELECT balance
INTO @current_balance
FROM wallets
WHERE user_id = 101
FOR UPDATE;

-- ২. Application-এ check করো (অথবা SQL-এ)
-- IF @current_balance < 400 THEN ROLLBACK; END IF;

-- ৩. Sufficient balance? Debit করো
UPDATE wallets
  SET
    balance      = balance - 400,
    updated_at  = NOW()
  WHERE user_id = 101
    AND balance >= 400;  -- extra safety

-- ৪. Affected rows check করো
-- ROW_COUNT() = 0 মানে balance insufficient

COMMIT;
  • Index ছাড়া WHERE clause দিয়ে FOR UPDATE — full table scan = table-level lock! পুরো table blocked হবে।
  • FOR UPDATE করে কিন্তু COMMIT/ROLLBACK না করা — lock indefinitely held, timeout পর্যন্ত সব blocked
  • Transaction-এর বাইরে FOR UPDATE — কোনো effect নেই, autocommit immediately release করে
05

LOCK IN SHARE MODE (FOR SHARE)

Shared Lock — পড়তে পারবে, কিন্তু কেউ পরিবর্তন করতে পারবে না

LOCK IN SHARE MODE (MySQL 8+ এ FOR SHARE) একটি shared lock নেয়। একাধিক transaction একই row-এ shared lock নিতে পারে, কিন্তু কেউ exclusive lock (FOR UPDATE বা UPDATE) নিতে পারবে না।

Lock Typeঅন্য Shared Lockঅন্য Exclusive Lockব্যবহার
FOR UPDATE (exclusive)❌ Block❌ BlockWrite করবো
LOCK IN SHARE MODE (shared)✅ OK❌ BlockRead করবো, write করবো না
SQL — LOCK IN SHARE MODE Syntax 
START TRANSACTION;

-- Shared lock — parent record exist করে কিনা verify করছি
SELECT id, stock
FROM products
WHERE id = 42
LOCK IN SHARE MODE;

-- এই transaction-এ product exist করে confirmed
-- এখন order insert করো — product কেউ delete করতে পারবে না
INSERT INTO order_items (order_id, product_id, qty)
VALUES (999, 42, 3);

COMMIT;

-- MySQL 8+ syntax
SELECT * FROM products WHERE id = 42 FOR SHARE;
📦 Real Scenario — Order ও Parent Record Consistency

E-commerce-এ order insert করার সময় product exist করে কিনা confirm করতে হয়। LOCK IN SHARE MODE নিলে — অন্য transaction product delete করতে পারবে না যতক্ষণ order insert complete না হয়।

Foreign key constraint থাকলেও race condition হতে পারে। Shared lock সেটা prevent করে।

⚠️ Deadlock সম্ভাবনা

দুটো transaction একই row-এ LOCK IN SHARE MODE নিয়ে তারপর দুজনেই FOR UPDATE চাইলে — deadlock! দুজনেই অপেক্ষা করবে। এই pattern এড়াও।

  • LOCK IN SHARE MODE নিয়ে তারপর UPDATE করার চেষ্টা — এটা deadlock-prone। সরাসরি FOR UPDATE নাও।
  • Report generation-এ LOCK IN SHARE MODE — অনেকক্ষণ lock held, writer blocked। Read-replica বা snapshot ব্যবহার করো।
06

FOR UPDATE SKIP LOCKED

Locked row skip করো — Job Queue-এর জাদুকরী tool

সাধারণত FOR UPDATE blocked row-এর জন্য wait করে। SKIP LOCKED দিলে — currently locked row গুলো skip করে পরেরটা নেয়। এটি multi-worker job processing-এর জন্য perfect।

SQL — SKIP LOCKED Basic Syntax 
START TRANSACTION;

-- Pending jobs থেকে একটা নাও (যেটা অন্য worker process করছে না)
SELECT id, payload, attempts
FROM jobs
WHERE status = 'pending'
  AND run_at <= NOW()
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- Job পেলে process করো ও status update করো
UPDATE jobs
  SET status = 'processing', started_at = NOW()
  WHERE id = ?;  -- application-এ id ব্যবহার করো

COMMIT;
⚙️ Real Scenario — Multi-Worker Email Queue

১০টি worker process একসাথে চলছে। প্রতিটি worker FOR UPDATE SKIP LOCKED দিয়ে query করে — প্রতিটি আলাদা job পাবে, কেউ একই job দুইবার process করবে না।

SKIP LOCKED ছাড়া? সবাই একই job-এর জন্য wait করতো (queue-এর মতো), বা same job multiple বার process হতো।

Complete Job Queue Implementation

SQL — Job Queue Table + Worker Pattern 
-- Job queue table
CREATE TABLE jobs (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  queue       VARCHAR(100) NOT NULL DEFAULT 'default',
  payload     JSON NOT NULL,
  status      ENUM('pending','processing','done','failed') DEFAULT 'pending',
  attempts    TINYINT DEFAULT 0,
  run_at      DATETIME DEFAULT NOW(),  -- delayed job support
  created_at  DATETIME DEFAULT NOW(),
  INDEX idx_queue_status (queue, status, run_at)
);

-- Worker: একটি job নাও এবং mark করো
START TRANSACTION;

SELECT id, payload
FROM jobs
WHERE queue   = 'emails'
  AND status  = 'pending'
  AND run_at  <= NOW()
  AND attempts < 3
ORDER BY run_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- Application এই row process করে, তারপর:
UPDATE jobs
  SET status   = 'done',
      attempts = attempts + 1
  WHERE id = :job_id;

COMMIT;
  • LIMIT ছাড়া SKIP LOCKED — সব unlocked rows return হবে, worker একসাথে হাজার job নিয়ে নেবে
  • Index না থাকলে SKIP LOCKED — full scan হয়, প্রতিটি locked row এক এক করে check করে, অনেক slow
  • Transaction commit না করে loop করা — job locked থাকে, অন্য worker কখনো পাবে না
07

FOR UPDATE NOWAIT

Wait না করে immediate fail — Fast-fail pattern

Normal FOR UPDATE lock পেতে wait করে (innodb_lock_wait_timeout পর্যন্ত, default 50 seconds)। NOWAIT দিলে lock পাওয়া না গেলে তাৎক্ষণিকভাবে error throw করে। User-facing API-তে এটা অনেক ভালো — user 50 সেকেন্ড wait করে না।

SQL — FOR UPDATE NOWAIT 
START TRANSACTION;

SELECT *
FROM seats
WHERE id = 55
  AND status = 'available'
FOR UPDATE NOWAIT;

-- Lock পেলে booking করো
-- Lock না পেলে: ERROR 3572 (ER_LOCK_NOWAIT)
-- Application তখন user-কে দেখাবে: "Seat currently being booked by another user"

COMMIT;
🎫 Real Scenario — Ticket Booking API

Concert ticket booking-এ user click করলে API call যায়। NOWAIT দিয়ে lock try করো — কেউ আগে নিয়ে থাকলে ৫০ সেকেন্ড wait না করে তাৎক্ষণিক "This seat is being booked" দেখাও। User experience অনেক ভালো।

PHP — NOWAIT Error Handling 
try {
    $pdo->beginTransaction();

    $stmt = $pdo->prepare("
        SELECT id, status FROM seats
        WHERE id = ? AND status = 'available'
        FOR UPDATE NOWAIT
    ");
    $stmt->execute([$seatId]);
    $seat = $stmt->fetch();

    if (!$seat) {
        $pdo->rollBack();
        return ['error' => 'Seat not available'];
    }

    $pdo->prepare("UPDATE seats SET status = 'booked', user_id = ? WHERE id = ?")
        ->execute([$userId, $seatId]);

    $pdo->commit();
    return ['success' => true];

} catch (PDOException $e) {
    $pdo->rollBack();

    // 3572 = ER_LOCK_NOWAIT — কেউ এই moment-এ booking করছে
    if ($e->getCode() == 3572) {
        return ['error' => 'Seat is being booked right now, try again'];
    }
    throw $e;
}
  • NOWAIT error handle না করা — user 500 error দেখবে, ব্যাখ্যা ছাড়া
  • Retry logic ছাড়া NOWAIT — legitimate concurrent request fail হবে, revenue হারাবে। Exponential backoff দিয়ে retry করো।

Race Condition

Race Condition কী এবং কিভাবে ধ্বংস করে?

Race condition তখন হয় যখন দুই বা তার বেশি concurrent operation একই data read করে, decision নেয়, তারপর write করে — কিন্তু একজনের write অন্যজনের read-এর পর কিন্তু write-এর আগে হয়। ফলে একজনের update হারিয়ে যায়।

মূল balance ৫০০। দুটো payment হয়েছে ৩০০+৪০০=৭০০ টাকার। কিন্তু final balance = ১০০ (শুধু B-এর deduction)। A-এর ৩০০ টাকার deduction হারিয়ে গেছে — user টাকা পেয়েছে কিন্তু wallet থেকে কাটেনি!

সমাধান — Atomic Update

SQL — Atomic UPDATE (Race Condition-safe) 
-- ❌ WRONG — Race condition সম্ভব
SELECT balance FROM wallets WHERE id = 1;
-- application-এ 500 - 300 = 200 calculate করো
UPDATE wallets SET balance = 200 WHERE id = 1;

-- ✅ CORRECT — Atomic, race condition-safe
UPDATE wallets
  SET balance = balance - 300
  WHERE id = 1
    AND balance >= 300;  -- insufficient balance guard

-- ROW_COUNT() = 0 মানে balance কম ছিল

-- ✅ CORRECT — FOR UPDATE দিয়ে (read-then-write pattern হলে)
START TRANSACTION;
SELECT balance FROM wallets WHERE id = 1 FOR UPDATE;
-- safe to read and compute
UPDATE wallets SET balance = balance - 300 WHERE id = 1;
COMMIT;
  • Application-এ balance calculate করে SET balance = new_value — সবচেয়ে বড় race condition pattern
  • Transaction ছাড়া read-then-write — ACID নেই, race guaranteed

Deadlock

Deadlock কি — দুই transaction একে অপরের জন্য অপেক্ষা করছে

Deadlock হয় যখন Transaction A, B-র lock-এর জন্য wait করছে, আর Transaction B, A-র lock-এর জন্য wait করছে — circular dependency। MySQL এটা detect করে একটিকে automatically kill করে (victim বানায়) এবং ERROR 1213: Deadlock found throw করে।

Classic Deadlock উদাহরণ

Transaction A — user 1 থেকে user 2
T1
FOR UPDATE on wallet_id=1 ✅ lock got
T3
FOR UPDATE on wallet_id=2 ⏳ WAITING (B holds it)
Transaction B — user 2 থেকে user 1
T2
FOR UPDATE on wallet_id=2 ✅ lock got
T4
FOR UPDATE on wallet_id=1 ⏳ WAITING (A holds it)
💀 Circular Wait = Deadlock

A waits for B's lock, B waits for A's lock → MySQL detects → kills one transaction (usually smaller one) → that transaction gets ERROR 1213.

Deadlock Prevention — Consistent Lock Ordering

SQL — Deadlock-Safe Fund Transfer 
-- ❌ WRONG — Order depend করে কে sender, কে receiver
-- Transaction A: lock wallet 1, then wallet 2
-- Transaction B: lock wallet 2, then wallet 1 → DEADLOCK!

-- ✅ CORRECT — সবসময় smaller ID আগে lock করো
-- Application-এ sort করো: MIN(from_id, to_id) আগে

START TRANSACTION;

-- Always lock lower ID first (prevents circular wait)
SELECT id, balance FROM wallets
WHERE id IN (1, 2)
ORDER BY id ASC  -- ← এটাই key! consistent ordering
FOR UPDATE;

UPDATE wallets SET balance = balance - 500 WHERE id = 1;
UPDATE wallets SET balance = balance + 500 WHERE id = 2;

COMMIT;

Deadlock হলে কী করবো?

PHP — Deadlock Retry Pattern 
function transferWithRetry($from, $to, $amount, $maxRetries = 3) {
    $attempt = 0;

    while ($attempt < $maxRetries) {
        try {
            $pdo->beginTransaction();

            // Lock in consistent order
            $ids = [$from, $to];
            sort($ids);

            $stmt = $pdo->prepare("
                SELECT id, balance FROM wallets
                WHERE id IN (?, ?)
                ORDER BY id ASC
                FOR UPDATE
            ");
            $stmt->execute($ids);

            // ... transfer logic ...

            $pdo->commit();
            return true;

        } catch (PDOException $e) {
            $pdo->rollBack();

            // 1213 = Deadlock, retry করো
            if ($e->getCode() == 1213 && $attempt < $maxRetries - 1) {
                $attempt++;
                usleep(rand(10000, 50000)); // 10-50ms random wait
                continue;
            }
            throw $e;
        }
    }
}

Row-Level vs Table-Level Locking

কখন row lock হয়, কখন table lock হয়?

InnoDB row-level locking করে — শুধু যে row(s) access করছো সেগুলো lock হয়। কিন্তু কিছু conditions-এ পুরো table lock হতে পারে, যেটা catastrophic।

Lock Typeকী lock হয়Concurrencyকখন হয়
Row Lock শুধু affected rows High — অন্য rows free Indexed WHERE clause, FOR UPDATE
Gap Lock দুই row-এর মধ্যবর্তী gap Medium Range query, REPEATABLE READ
Next-Key Lock Row + preceding gap Medium InnoDB default range locking
Table Lock পুরো table Low — সব blocked No index, full scan, DDL, MyISAM
SQL — Row Lock vs Table Lock (Index matters!) 
-- ✅ ROW LOCK — id is primary key (indexed)
SELECT * FROM products WHERE id = 42 FOR UPDATE;
-- শুধু row id=42 locked. বাকি সব rows free.

-- ⚠️ TABLE LOCK RISK — non-indexed column
SELECT * FROM products WHERE name = 'Widget' FOR UPDATE;
-- 'name'-এ index না থাকলে: full table scan = table-level lock!
-- পুরো products table locked হবে, সব operations blocked!

-- Index আছে কিনা দেখো
EXPLAIN SELECT * FROM products WHERE name = 'Widget' FOR UPDATE\G
-- type = 'ALL' মানে full scan = table lock risk

-- সমাধান: index যোগ করো
ALTER TABLE products ADD INDEX idx_name (name);

-- Explicit table lock (প্রয়োজনে)
LOCK TABLES products WRITE;
-- ... bulk operation ...
UNLOCK TABLES;
⚠️ LOCK TABLES vs InnoDB Transactions

LOCK TABLES InnoDB transaction-এর সাথে mix করা dangerous। LOCK TABLES করলে current transaction implicit COMMIT হয়। Production-এ সাধারণত SELECT FOR UPDATE ব্যবহার করো।

GET_LOCK() এবং RELEASE_LOCK()

Application-level Named Lock — যেকোনো resource lock করো

GET_LOCK() হলো MySQL-এর advisory lock system। এটি database row-এর সাথে সম্পর্কিত না — তুমি যেকোনো string দিয়ে একটি named lock তৈরি করতে পারো। এটি distributed locking-এর জন্য useful যেখানে multiple application server একটি shared resource access করছে।

SQL — GET_LOCK / RELEASE_LOCK Syntax 
-- Lock নাও (timeout = 10 seconds)
-- Returns: 1 (got lock), 0 (timeout), NULL (error)
SELECT GET_LOCK('payment_user_101', 10);

-- Lock আছে কিনা check করো (non-blocking)
SELECT IS_FREE_LOCK('payment_user_101');    -- 1=free, 0=locked
SELECT IS_USED_LOCK('payment_user_101');   -- connection id বা NULL

-- Lock ছেড়ে দাও
SELECT RELEASE_LOCK('payment_user_101');   -- 1=released, 0=not held

-- সব lock ছাড়ো
SELECT RELEASE_ALL_LOCKS();
💳 Real Scenario — User Payment Concurrency Control

একই user-এর একসাথে দুটো payment request আসলে — পুরো payment flow lock করো। Wallet row-এ FOR UPDATE ছাড়াও GET_LOCK দিয়ে user-level lock নেওয়া যায়। এটি complex multi-table operation-এ useful।

PHP — GET_LOCK Pattern for Payment 
function processPayment($userId, $amount) {
    $lockName = "payment_user_{$userId}";

    // Lock নাও (10 second timeout)
    $result = $pdo->query("SELECT GET_LOCK('{$lockName}', 10)")->fetchColumn();

    if ($result != 1) {
        throw new Exception("Could not acquire payment lock. Please try again.");
    }

    try {
        $pdo->beginTransaction();

        // Check balance
        $balance = $pdo->query("SELECT balance FROM wallets WHERE user_id = {$userId} FOR UPDATE")
                       ->fetchColumn();

        if ($balance < $amount) {
            throw new Exception("Insufficient balance");
        }

        // Debit
        $pdo->exec("UPDATE wallets SET balance = balance - {$amount} WHERE user_id = {$userId}");

        // Log transaction
        $pdo->exec("INSERT INTO transactions (user_id, amount, type) VALUES ({$userId}, {$amount}, 'debit')");

        $pdo->commit();
        return true;

    } catch (Exception $e) {
        $pdo->rollBack();
        throw $e;
    } finally {
        // Always release the lock!
        $pdo->query("SELECT RELEASE_LOCK('{$lockName}')");
    }
}
📌 GET_LOCK-এর সীমাবদ্ধতা

GET_LOCK connection-specific। Connection বন্ধ হলে lock automatically release হয়। Connection pool-এ সাবধান — একই connection reuse হলে পুরনো lock থাকতে পারে। Production-এ Redis-based distributed lock (RedLock) অনেক সময় ভালো।

  • GET_LOCK করে RELEASE_LOCK না করা — সেই connection যতক্ষণ open, lock held। Next request same connection পেলে conflict।
  • Lock name collision — "user_1" কি user_id=1 নাকি product_id=1? Prefix দাও: "payment:user:101"
  • Timeout=0 (no wait) — busy system-এ প্রায়ই fail করবে। Meaningful timeout দাও।

INSERT … ON DUPLICATE KEY UPDATE

Upsert — Insert অথবা Update, একটি query-তে

INSERT ... ON DUPLICATE KEY UPDATE হলো atomic upsert operation। Primary key বা UNIQUE key conflict হলে INSERT এর বদলে UPDATE চালায়। Separate SELECT + INSERT/UPDATE করার দরকার নেই, race condition নেই।

SQL — ON DUPLICATE KEY UPDATE Syntax 
-- Table structure
CREATE TABLE product_views (
  product_id  INT NOT NULL,
  user_id     INT NOT NULL,
  view_count  INT DEFAULT 1,
  last_viewed DATETIME DEFAULT NOW(),
  PRIMARY KEY (product_id, user_id)
);

-- Upsert: নতুন হলে insert, আগে থেকে থাকলে update
INSERT INTO product_views (product_id, user_id, view_count, last_viewed)
VALUES (42, 101, 1, NOW())
ON DUPLICATE KEY UPDATE
  view_count  = view_count + 1,
  last_viewed = NOW();

-- VALUES() function দিয়ে insert value reference করো
INSERT INTO inventory (product_id, stock)
VALUES (42, 100)
ON DUPLICATE KEY UPDATE
  stock = stock + VALUES(stock);
📊 Real Scenario — Counter / Stats Table

High-traffic site-এ page view count track করা। SELECT → if exists UPDATE else INSERT করলে race condition। ON DUPLICATE KEY UPDATE একটি atomic operation — race-free, fast, simple।

Inventory Stock Adjustment

SQL — Inventory Upsert Pattern 
-- Warehouse থেকে stock receive করা — upsert pattern
INSERT INTO inventory (product_id, warehouse_id, quantity, last_updated)
VALUES
  (101, 1, 50, NOW()),
  (102, 1, 30, NOW()),
  (103, 1, 100, NOW())
ON DUPLICATE KEY UPDATE
  quantity     = quantity + VALUES(quantity),
  last_updated = VALUES(last_updated);

-- MySQL 8.0.19+ এ cleaner syntax:
INSERT INTO inventory (product_id, quantity)
VALUES (101, 50) AS new
ON DUPLICATE KEY UPDATE
  quantity = inventory.quantity + new.quantity;
  • UNIQUE key ছাড়া ON DUPLICATE KEY UPDATE — কখনো trigger হবে না, সবসময় insert হবে
  • Multiple unique keys — যেকোনো একটিতে conflict হলে UPDATE trigger হয়, unexpected behavior
  • AUTO_INCREMENT counter-এ — ON DUPLICATE KEY UPDATE হলেও auto_increment বাড়তে থাকে, gaps তৈরি হয়

  to_user_id      BIGINT UNSIGNED NOT NULL,

SQL — Complete Payment Transaction 
-- Step 1: Idempotency check (duplicate payment prevention)
INSERT INTO payments (idempotency_key, from_user_id, to_user_id, amount)
VALUES ('txn_abc123', 1, 2, 500.00)
ON DUPLICATE KEY UPDATE id = id;  -- no-op on duplicate
-- ROW_COUNT() = 2 means new, 0 means duplicate → return cached result

START TRANSACTION;

-- Step 2: Lock both wallets (lower ID first to prevent deadlock)
SELECT id, user_id, balance
FROM wallets
WHERE user_id IN (1, 2)
ORDER BY id ASC
FOR UPDATE;

-- Step 3: Validate balance (application-side check)
-- IF sender_balance < 500 THEN ROLLBACK; END IF;

-- Step 4: Atomic debit + credit
UPDATE wallets
  SET balance = balance - 500.00
  WHERE user_id = 1 AND balance >= 500.00;

UPDATE wallets
  SET balance = balance + 500.00
  WHERE user_id = 2;

-- Step 5: Ledger entries
INSERT INTO wallet_ledger (wallet_id, payment_id, type, amount, balance)
VALUES
  (1, @payment_id, 'debit',  500.00, @sender_new_balance),
  (2, @payment_id, 'credit', 500.00, @receiver_new_balance);

-- Step 6: Update payment status
UPDATE payments SET status = 'completed' WHERE id = @payment_id;

COMMIT;

Real System: Inventory Management

Stock Overselling Prevention

E-commerce-এর সবচেয়ে বড় nightmare হলো overselling — ১০টি stock আছে কিন্তু ১৫টি order নিয়ে নেওয়া। Proper locking ছাড়া এটা সহজেই হয়।

SQL — Safe Stock Deduction 
START TRANSACTION;

-- Lock the product row first
SELECT id, name, stock, reserved
FROM products
WHERE id = 42
FOR UPDATE;

-- Application: available = stock - reserved
-- IF available < order_qty THEN ROLLBACK; END IF;

-- Atomic stock deduction (double safety with WHERE clause)
UPDATE products
  SET
    stock    = stock - 2,
    sold     = sold + 2
  WHERE id = 42
    AND stock >= 2;  -- guard: sufficient stock

-- ROW_COUNT() = 0 মানে stock কমে গেছে, order fail
SELECT ROW_COUNT() INTO @affected;
-- IF @affected = 0 THEN ROLLBACK; SIGNAL "Out of stock"; END IF;

-- Order create করো
INSERT INTO orders (user_id, product_id, qty, status)
VALUES (101, 42, 2, 'confirmed');

COMMIT;

-- ══ OPTIMISTIC LOCKING Alternative ══
-- Version-based, no explicit lock, retry on conflict
UPDATE products
  SET
    stock   = stock - 2,
    version = version + 1
  WHERE id      = 42
    AND version = 7    -- previously read version
    AND stock   >= 2;
-- ROW_COUNT() = 0: someone else updated → retry

Real System: Job Queue Workers

Database-backed Job Queue — Multi-Worker Safe

Laravel Queue, Sidekiq-style database job queue নিজে implement করতে পারো SKIP LOCKED দিয়ে।

SQL — Distributed Job Queue (SKIP LOCKED) 
-- Worker process: একটি job claim করো
START TRANSACTION;

SELECT id, type, payload, attempts
FROM jobs
WHERE queue     = 'default'
  AND status    = 'pending'
  AND run_at    <= NOW()
  AND attempts  < max_attempts
ORDER BY priority DESC, run_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- Job পেলে: claim করো
UPDATE jobs
  SET
    status       = 'processing',
    attempts     = attempts + 1,
    started_at   = NOW(),
    worker_id    = @worker_id,
    reserved_until = DATE_ADD(NOW(), INTERVAL 5 MINUTE)
  WHERE id = @job_id;

COMMIT;

-- Process করার পর: complete করো
UPDATE jobs
  SET status = 'completed', completed_at = NOW()
  WHERE id = @job_id;

-- Fail হলে: retry schedule করো
UPDATE jobs
  SET
    status  = IF(attempts >= max_attempts, 'failed', 'pending'),
    run_at  = DATE_ADD(NOW(), INTERVAL POW(2, attempts) MINUTE),  -- exponential backoff
    last_error = @error_message
  WHERE id = @job_id;

-- Stuck jobs recover (worker crashed, reserved_until expired)
UPDATE jobs
  SET status = 'pending', worker_id = NULL
  WHERE status = 'processing'
    AND reserved_until < NOW();

Real System: Booking System

Double Booking Prevention — Seat, Room, Slot

Concert seat, hotel room, doctor appointment — এই সব booking system-এ double booking prevent করতে হয়।

SQL — Seat Booking with FOR UPDATE NOWAIT 
CREATE TABLE seats (
  id          INT PRIMARY KEY,
  event_id    INT NOT NULL,
  section     VARCHAR(10),
  row         CHAR(2),
  number      INT,

-- User selects a seat → temporary hold (5 min to complete payment)

  WHERE id = 55
    AND status = 'available';

COMMIT;

-- Payment complete → confirm booking
START TRANSACTION;

SELECT id FROM seats
WHERE id = 55
  AND status     = 'held'
  AND held_by    = 101
  AND held_until > NOW()
FOR UPDATE;

UPDATE seats
  SET status = 'booked', held_until = NULL
  WHERE id = 55;

Real System: Financial Ledger

Double-Entry Accounting — Immutable Ledger

Financial system-এ balance directly update না করে ledger entries দিয়ে balance calculate করা হয়। প্রতিটি transaction দুটো entry (debit + credit) — double-entry bookkeeping।

SQL — Append-Only Ledger Pattern 
CREATE TABLE ledger_entries (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  account_id  BIGINT UNSIGNED NOT NULL,
  txn_ref     VARCHAR(64) NOT NULL,  -- transaction reference
  entry_type  ENUM('DEBIT','CREDIT') NOT NULL,
  amount      DECIMAL(18,4) NOT NULL,
  currency    CHAR(3) DEFAULT 'BDT',
  description VARCHAR(255),
  created_at  DATETIME(6) DEFAULT NOW(6),  -- microsecond precision
  INDEX (account_id, created_at),
  INDEX (txn_ref)
  -- No UPDATE, No DELETE — append-only!
) ENGINE=InnoDB;

-- Balance = SUM(CREDIT) - SUM(DEBIT)
SELECT
  account_id,
  SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END)
  -
  SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE 0 END)
  AS balance
FROM ledger_entries
WHERE account_id = 101;

-- Performance: materialized balance cache (separate table)
CREATE TABLE account_balances (
  account_id  BIGINT UNSIGNED PRIMARY KEY,
  balance     DECIMAL(18,4) NOT NULL DEFAULT 0,
  last_entry_id BIGINT UNSIGNED,  -- sync pointer
  updated_at  DATETIME(6)
);

-- Insert ledger entry + update balance cache atomically
START TRANSACTION;

SELECT balance FROM account_balances
WHERE account_id = 101
FOR UPDATE;

-- Validate sufficient balance for debit...

INSERT INTO ledger_entries (account_id, txn_ref, entry_type, amount, description)
VALUES (101, 'TXN-2024-001', 'DEBIT', 500.00, 'Payment to vendor');

UPDATE account_balances
  SET
    balance       = balance - 500.00,
    last_entry_id = LAST_INSERT_ID(),
    updated_at    = NOW(6)
  WHERE account_id = 101;

COMMIT;

High-Concurrency Database Design Patterns

Optimistic vs Pessimistic Locking — কোনটা কখন?

Pattern কিভাবে কাজ করে ব্যবহার করো যখন Avoid করো যখন
Pessimistic Lock আগে lock নাও, তারপর কাজ করো (FOR UPDATE) Conflict likely, financial transactions Low contention, read-heavy
Optimistic Lock Version check দিয়ে update করো, fail হলে retry Low contention, read-heavy scenarios High conflict, cannot afford retries
Compare and Swap WHERE clause-এ old value check করো Simple counter update Complex multi-step operations
SQL — Optimistic Locking with Version 
-- ১. Read with version
SELECT id, stock, version
FROM products
WHERE id = 42;
-- Returns: stock=50, version=3

-- ২. Update with version check (no explicit lock needed)
UPDATE products
  SET
    stock   = stock - 5,
    version = version + 1
  WHERE id      = 42
    AND version = 3     -- stale check
    AND stock   >= 5;

-- ROW_COUNT() = 0 মানে:
-- অন্য কেউ আগে update করেছে → re-read করো, retry করো

-- ══ Counter Pattern (no lock needed) ══
-- Atomic increment — race condition-safe without transaction
UPDATE counters
  SET value = value + 1
  WHERE name = 'daily_signups';
-- MySQL UPDATE is atomic at row level

SQL — Sharded Counter (High Write Throughput) 
-- Problem: একটি row-এ লাখ লাখ concurrent update → hot spot
-- Solution: multiple "slot" rows, random slot update করো
CREATE TABLE distributed_counters (
  name   VARCHAR(50),
  slot   TINYINT,            -- 0-9, 10 slots
  count  BIGINT DEFAULT 0,
  PRIMARY KEY (name, slot)
);

-- Write: random slot-এ increment করো
-- Application: slot = rand(0,9)
INSERT INTO distributed_counters (name, slot, count)
VALUES ('page_views', 3, 1)
ON DUPLICATE KEY UPDATE count = count + 1;

-- Read: সব slots sum করো
SELECT SUM(count) AS total
FROM distributed_counters
WHERE name = 'page_views';
-- 10x throughput improvement!

সবচেয়ে Common Production Problems

  • Long transaction + user interaction: Transaction open রেখে HTTP response-এর জন্য wait বা user input নেওয়া — lock লম্বা সময় held থাকে। Solution: Transaction শুধু DB work-এর জন্য, শুরুতে সব input validate করো।
  • ORM-generated queries without index: Eloquent/Hibernate অনেক সময় un-indexed column-এ WHERE দেয়, FOR UPDATE হলে table lock। সবসময় EXPLAIN দিয়ে verify করো।
  • Connection pool + GET_LOCK: Connection pool-এ পুরনো connection reuse হলে পুরনো lock থাকে। Always RELEASE_LOCK() করো, finally block-এ।
  • Replica lag + read-after-write: Write করে immediately replica থেকে read — stale data। Critical read সবসময় primary থেকে করো, বা write-after-read pattern avoid করো।
  • Bulk operations without batching: DELETE FROM logs WHERE created_at < '2023-01-01' — কোটি row lock হয়, production hang। Batch করো: LIMIT 1000, loop করো।
SQL — Batch Delete (Lock-friendly) 
-- ❌ DANGEROUS — কোটি rows একসাথে lock
DELETE FROM logs WHERE created_at < '2023-01-01';

-- ✅ SAFE — Batch করো, pause করো
REPEAT
  DELETE FROM logs
  WHERE created_at < '2023-01-01'
  LIMIT 1000;

  SET @deleted = ROW_COUNT();
  DO SLEEP(0.1);  -- 100ms pause

UNTIL @deleted = 0
END REPEAT;

-- Monitoring: lock wait timeout configure
SET GLOBAL innodb_lock_wait_timeout = 10;   -- সব connection
SET SESSION innodb_lock_wait_timeout = 3;  -- এই session

-- Performance schema enable করো
UPDATE performance_schema.setup_instruments
  SET ENABLED = 'YES'
  WHERE NAME LIKE 'wait/lock/innodb%';
