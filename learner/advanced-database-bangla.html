01 🧱

Database মৌলিক ধারণা ও RDBMS

Database হলো সংগঠিতভাবে তথ্য সংরক্ষণের পদ্ধতি। RDBMS (Relational Database Management System) হলো সেই ব্যবস্থা যেখানে তথ্য Table আকারে এবং Relation দিয়ে সংযুক্ত থাকে।

📊 Table (সারণি)

Row (রেকর্ড) ও Column (ক্ষেত্র) দিয়ে তৈরি কাঠামো। প্রতিটি table একটি entity বোঝায়।

🔑 Primary Key

প্রতিটি রেকর্ডকে uniquely চিহ্নিত করে। Null বা Duplicate হতে পারে না।

🔗 Foreign Key

অন্য table-এর Primary Key-কে refer করে। দুটি table-এর মধ্যে সম্পর্ক তৈরি করে।

02📐

Normalization — ১NF থেকে BCNF

Normalization হলো Database-কে এমনভাবে সাজানো যাতে তথ্যের পুনরাবৃত্তি (Redundancy) কমে এবং Anomaly (ভুল সমস্যা) না হয়।

Anomaly তিন ধরনের: Insert Anomaly (নতুন তথ্য যোগে সমস্যা), Update Anomaly (একটি পরিবর্তন করলে সব জায়গায় করতে হয়), Delete Anomaly (মুছলে অন্য তথ্যও হারিয়ে যায়)।

1NF — প্রথম স্বাভাবিক রূপ

নিয়ম: প্রতিটি Column-এ একটি মাত্র মান থাকবে। কোনো Repeating Group থাকবে না।

❌ 1NF ভাঙছে

OrderIDProducts
101চাল, ডাল, তেল
102আটা, চিনি

Products column-এ একাধিক মান আছে!

✅ 1NF সঠিক

OrderIDProduct
101চাল
101ডাল
101তেল
102আটা

2NF — দ্বিতীয় স্বাভাবিক রূপ

নিয়ম: 1NF হতে হবে এবং প্রতিটি non-key column সম্পূর্ণ Primary Key-এর উপর নির্ভরশীল হতে হবে (Partial Dependency নেই)।

SQL — 2NF প্রয়োগ
-- ❌ 2NF ভাঙছে: CustomerName শুধু CustomerID-এর উপর নির্ভরশীল
-- OrderItems(OrderID, ProductID, Qty, CustomerName, ProductName)

-- ✅ সমাধান: আলাদা table ভাগ করুন
CREATE TABLE orders (
  order_id    INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT NOT NULL,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_items (
  id         INT PRIMARY KEY AUTO_INCREMENT,
  order_id   INT NOT NULL,
  product_id INT NOT NULL,
  quantity   INT NOT NULL,
  price      DECIMAL(10,2) NOT NULL,
  FOREIGN KEY (order_id)   REFERENCES orders(order_id),
  FOREIGN KEY (product_id) REFERENCES products(product_id)
);

3NF — তৃতীয় স্বাভাবিক রূপ

নিয়ম: 2NF হতে হবে এবং কোনো non-key column অন্য non-key column-এর উপর নির্ভরশীল হতে পারবে না (Transitive Dependency নেই)।

SQL — 3NF সমাধান
-- ❌ Employee(EmpID, DeptID, DeptName)
-- DeptName → DeptID → EmpID (Transitive Dependency!)

-- ✅ সমাধান: Department আলাদা করুন
CREATE TABLE departments (
  dept_id   INT PRIMARY KEY,
  dept_name VARCHAR(100) NOT NULL,
  location  VARCHAR(100)
);

CREATE TABLE employees (
  emp_id   INT PRIMARY KEY AUTO_INCREMENT,
  name     VARCHAR(100) NOT NULL,
  email    VARCHAR(150) UNIQUE NOT NULL,
  dept_id  INT,
  salary   DECIMAL(12,2),
  FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);

BCNF — Boyce-Codd Normal Form

🎯 BCNF নিয়ম

3NF-এর কঠোর সংস্করণ। প্রতিটি determinant অবশ্যই candidate key হতে হবে। ব্যবহারিকভাবে 3NF পর্যন্ত পৌঁছালেই বেশিরভাগ সমস্যা সমাধান হয়।

Normal Formমূল নিয়মকী দূর হয়
1NFAtomic values, কোনো repeating group নেইMulti-valued column
2NF1NF + কোনো Partial Dependency নেইআংশিক নির্ভরতা
3NF2NF + কোনো Transitive Dependency নেইপরোক্ষ নির্ভরতা
BCNF3NF + প্রতিটি determinant candidate keyবিশেষ anomaly

সম্পূর্ণ Schema তৈরি

SQL — ecommerce_db
-- ১. Categories Table
CREATE TABLE categories (
  category_id   INT PRIMARY KEY AUTO_INCREMENT,
  name          VARCHAR(100) NOT NULL UNIQUE,
  parent_id     INT DEFAULT NULL,       -- Self-referencing (Sub-category)
  slug          VARCHAR(120) NOT NULL UNIQUE,
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (parent_id) REFERENCES categories(category_id)
    ON DELETE SET NULL
) ENGINE=InnoDB;

-- ২. Users Table
CREATE TABLE users (
  user_id       INT PRIMARY KEY AUTO_INCREMENT,
  name          VARCHAR(100) NOT NULL,
  email         VARCHAR(150) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  phone         VARCHAR(20),
  role          ENUM('customer','admin','vendor') DEFAULT 'customer',
  is_verified   TINYINT(1) DEFAULT 0,
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;

-- ৩. Products Table
CREATE TABLE products (
  product_id    INT PRIMARY KEY AUTO_INCREMENT,
  category_id   INT NOT NULL,
  name          VARCHAR(200) NOT NULL,
  slug          VARCHAR(220) UNIQUE NOT NULL,
  description   TEXT,
  price         DECIMAL(12,2) NOT NULL,
  sale_price    DECIMAL(12,2),
  stock         INT NOT NULL DEFAULT 0,
  sku           VARCHAR(50) UNIQUE,
  weight_kg     DECIMAL(6,3),
  attributes    JSON,               -- {"color":"red","size":"XL"}
  is_active     TINYINT(1) DEFAULT 1,
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (category_id) REFERENCES categories(category_id)
) ENGINE=InnoDB;

-- ৪. Orders Table
CREATE TABLE orders (
  order_id      INT PRIMARY KEY AUTO_INCREMENT,
  user_id       INT NOT NULL,
  total_amount  DECIMAL(12,2) NOT NULL,
  discount      DECIMAL(10,2) DEFAULT 0,
  status        ENUM('pending','processing','shipped','delivered','cancelled')
                DEFAULT 'pending',
  payment_method VARCHAR(50),
  shipping_address JSON,           -- ঠিকানা JSON হিসেবে
  notes         TEXT,
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE RESTRICT
) ENGINE=InnoDB;

-- ৫. Order Items
CREATE TABLE order_items (
  item_id       INT PRIMARY KEY AUTO_INCREMENT,
  order_id      INT NOT NULL,
  product_id    INT NOT NULL,
  quantity      INT NOT NULL,
  unit_price    DECIMAL(12,2) NOT NULL, -- অর্ডারের সময়ের মূল্য
  FOREIGN KEY (order_id)   REFERENCES orders(order_id)   ON DELETE CASCADE,
  FOREIGN KEY (product_id) REFERENCES products(product_id) ON DELETE RESTRICT
) ENGINE=InnoDB;

Advanced MySQL — Joins, Subqueries, Window Functions

সব ধরনের JOIN

SQL — Joins
-- INNER JOIN: দুটো table-এ মিলে যাওয়া রেকর্ড
SELECT u.name, o.order_id, o.total_amount
FROM users u
INNER JOIN orders o ON u.user_id = o.user_id;

-- LEFT JOIN: সব user দেখাবে, order না থাকলেও
SELECT u.name, COUNT(o.order_id) AS total_orders
FROM users u
LEFT JOIN orders o ON u.user_id = o.user_id
GROUP BY u.user_id;

-- MULTI-TABLE JOIN: তিনটি table একসাথে
SELECT
  u.name     AS customer,
  p.name     AS product,
  oi.quantity,
  oi.unit_price,
  (oi.quantity * oi.unit_price) AS subtotal
FROM users u
JOIN orders o       ON u.user_id   = o.user_id
JOIN order_items oi ON o.order_id  = oi.order_id
JOIN products p     ON oi.product_id = p.product_id
WHERE o.status = 'delivered'
ORDER BY o.created_at DESC;

Subquery ও CTE (Common Table Expression)

SQL — CTE & Subquery
-- CTE: জটিল query-কে সহজে পড়া যায়
WITH top_customers AS (
  SELECT
    user_id,
    SUM(total_amount) AS lifetime_value,
    COUNT(*) AS order_count
  FROM orders
  WHERE status != 'cancelled'
  GROUP BY user_id
  HAVING lifetime_value > 10000
),
ranked AS (
  SELECT
    tc.*,
    u.name, u.email,
    RANK() OVER (ORDER BY lifetime_value DESC) AS rank_pos
  FROM top_customers tc
  JOIN users u ON tc.user_id = u.user_id
)
SELECT * FROM ranked WHERE rank_pos <= 10;

-- Correlated Subquery: প্রতিটি product-এর সর্বোচ্চ বিক্রির দিন
SELECT
  p.name,
  p.price,
  (
    SELECT SUM(oi.quantity)
    FROM order_items oi
    WHERE oi.product_id = p.product_id
  ) AS total_sold
FROM products p
ORDER BY total_sold DESC
LIMIT 10;

Window Functions — উন্নত বিশ্লেষণ

SQL — Window Functions
-- ROW_NUMBER, RANK, DENSE_RANK
SELECT
  name,
  salary,
  dept_id,
  ROW_NUMBER()  OVER (PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
  RANK()        OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk,
  DENSE_RANK()  OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dense_rnk
FROM employees;

-- Running Total (চলমান যোগফল)
SELECT
  order_id,
  total_amount,
  SUM(total_amount) OVER (
    ORDER BY created_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM orders;

-- LAG/LEAD: আগের/পরের রেকর্ড তুলনা
SELECT
  DATE(created_at) AS sale_date,
  SUM(total_amount) AS daily_sales,
  LAG(SUM(total_amount)) OVER (ORDER BY DATE(created_at)) AS prev_day,
  SUM(total_amount) - LAG(SUM(total_amount)) OVER (ORDER BY DATE(created_at))
    AS growth
FROM orders
GROUP BY DATE(created_at);

Index তৈরির নিয়ম

SQL — Indexes
-- সাধারণ Index
CREATE INDEX idx_user_email ON users(email);

-- Composite Index (যৌথ): column-এর ক্রম গুরুত্বপূর্ণ!
-- Rule: WHERE clause-এর ক্রম অনুযায়ী column সাজান
CREATE INDEX idx_order_user_status ON orders(user_id, status, created_at);

-- Unique Index
CREATE UNIQUE INDEX idx_product_sku ON products(sku);

-- Full-text Index: বাংলা/ইংরেজি text search-এর জন্য
CREATE FULLTEXT INDEX idx_product_search ON products(name, description);

-- Full-text search ব্যবহার
SELECT product_id, name, price,
  MATCH(name, description) AGAINST('wireless headphone' IN BOOLEAN MODE) AS relevance
FROM products
WHERE MATCH(name, description) AGAINST('wireless headphone' IN BOOLEAN MODE)
ORDER BY relevance DESC;

-- JSON Column-এ Index (MySQL 8+)
ALTER TABLE products ADD COLUMN
  color VARCHAR(50) GENERATED ALWAYS AS (attributes->>'$.color') STORED;
CREATE INDEX idx_product_color ON products(color);

-- বিদ্যমান Index দেখা
SHOW INDEX FROM products;
SHOW INDEX FROM orders;

Transactions ও ACID Properties

Transaction হলো এক বা একাধিক SQL statement-এর সমষ্টি যা একসাথে সফল হয় অথবা একসাথে ব্যর্থ হয়।

A — Atomicity

সবকিছু হবে অথবা কিছুই হবে না।

C — Consistency

Database সবসময় valid state-এ থাকবে।

I — Isolation

একটি Transaction অন্যটিকে প্রভাবিত করবে না।

D — Durability

Commit হলে তথ্য স্থায়ীভাবে সংরক্ষিত হবে।

Bank Transfer — Transaction উদাহরণ

SQL — Transaction
-- ব্যাংক Transfer: রহিম → করিম = ৫০০০ টাকা
START TRANSACTION;

BEGIN;

  -- রহিমের account থেকে কাটা
  UPDATE accounts
  SET balance = balance - 5000
  WHERE account_id = 1001 AND balance >= 5000;

  -- Row affected কিনা চেক
  IF ROW_COUNT() = 0 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'ব্যালেন্স যথেষ্ট নেই';
  END IF;

  -- করিমের account-এ যোগ করা
  UPDATE accounts
  SET balance = balance + 5000
  WHERE account_id = 1002;

  -- লেনদেনের রেকর্ড রাখা
  INSERT INTO transactions (from_acc, to_acc, amount, txn_type)
  VALUES (1001, 1002, 5000, 'transfer');

COMMIT; -- সব সফল হলে স্থায়ী করুন

-- Error হলে সব ফিরিয়ে দিন
-- ROLLBACK;

Isolation Levels — বিচ্ছিন্নতার স্তর

LevelDirty ReadNon-repeatable ReadPhantom ReadPerformance
READ UNCOMMITTEDহয়হয়হয়দ্রুততম
READ COMMITTEDহয় নাহয়হয়দ্রুত
REPEATABLE READ (default)হয় নাহয় নাহয়মাঝারি
SERIALIZABLEহয় নাহয় নাহয় নাধীরতম
SQL — Isolation Level
-- Session-এর জন্য Isolation Level পরিবর্তন
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Pessimistic Locking: অন্যরা পড়তেও পারবে না
SELECT * FROM products
WHERE product_id = 5
FOR UPDATE; -- Row lock করা হলো

-- Optimistic Locking: version column দিয়ে
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE product_id = 5 AND version = 3; -- version মিলতে হবে

DELIMITER $$

CREATE PROCEDURE PlaceOrder(
  IN  p_user_id    INT,
  IN  p_product_id INT,
  IN  p_quantity   INT,
  OUT p_order_id   INT,
  OUT p_message    VARCHAR(200)
)
BEGIN
  DECLARE v_stock      INT;
  DECLARE v_price      DECIMAL(12,2);
  DECLARE v_total      DECIMAL(12,2);
  DECLARE EXIT HANDLER FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    SET p_message = 'ত্রুটি ঘটেছে, অর্ডার বাতিল';
  END;

  -- Stock ও price চেক
  SELECT stock, price INTO v_stock, v_price
  FROM products WHERE product_id = p_product_id FOR UPDATE;

  IF v_stock < p_quantity THEN
    SET p_message = 'পর্যাপ্ত stock নেই';
    LEAVE proc_label;
  END IF;

  SET v_total = v_price * p_quantity;

  START TRANSACTION;

  -- Order তৈরি
  INSERT INTO orders (user_id, total_amount, status)
  VALUES (p_user_id, v_total, 'pending');
  SET p_order_id = LAST_INSERT_ID();

  -- Order item যোগ
  INSERT INTO order_items (order_id, product_id, quantity, unit_price)
  VALUES (p_order_id, p_product_id, p_quantity, v_price);

  -- Stock কমানো
  UPDATE products SET stock = stock - p_quantity
  WHERE product_id = p_product_id;

  COMMIT;
  SET p_message = 'অর্ডার সফলভাবে তৈরি হয়েছে';
END$$

DELIMITER ;

-- Procedure চালানো
CALL PlaceOrder(1, 5, 2, @order_id, @msg);
SELECT @order_id, @msg;

Master-Slave Replication

Replication মানে এক Server-এর ডেটা অন্য Server-এ স্বয়ংক্রিয়ভাবে কপি হওয়া। Read performance বাড়ানো ও Backup-এর জন্য ব্যবহার করা হয়।

📝 Master Server

  • সব Write (INSERT/UPDATE/DELETE) এখানে
  • Binary Log তৈরি করে
  • Production App এখানে লেখে

📖 Slave Server(s)

  • শুধু Read (SELECT) করে
  • Master থেকে Replicate করে
  • Report ও Analytics এখানে

Database Sharding — Horizontal Scaling

Sharding মানে একটি বড় Database-কে ছোট ছোট টুকরায় ভাগ করে একাধিক Server-এ রাখা। Replication-এ সব server-এ একই data থাকে, Sharding-এ আলাদা আলাদা data থাকে।

📌 কখন Sharding দরকার?

  • Single server-এ আর write scale হচ্ছে না
  • Table-এ ১ বিলিয়নের বেশি Row
  • RAM-এ Index ধরছে না
  • প্রতিদিন কোটি কোটি নতুন রেকর্ড
JavaScript — Sharding Logic
const shards = [
  mysql.createPool({ host: 'shard-01.db', database: 'ecom_shard_0' }),
  mysql.createPool({ host: 'shard-02.db', database: 'ecom_shard_1' }),
  mysql.createPool({ host: 'shard-03.db', database: 'ecom_shard_2' }),
  mysql.createPool({ host: 'shard-04.db', database: 'ecom_shard_3' }),
];

// Hash-based Sharding: user_id দিয়ে কোন shard?
function getShard(userId) {
  return shards[userId % shards.length];
}

async function getUserOrders(userId) {
  const shard = getShard(userId);
  const [rows] = await shard.query(
    'SELECT * FROM orders WHERE user_id = ?', [userId]
  );
  return rows;
}

// Cross-shard query: সব shard থেকে aggregate
async function getTotalRevenue() {
  const promises = shards.map(shard =>
    shard.query('SELECT SUM(total_amount) AS rev FROM orders')
  );
  const results = await Promise.all(promises);
  return results.reduce((sum, [rows]) => sum + (rows[0].rev || 0), 0);
}

const bcrypt = require('bcrypt');

async function hashPassword(plainText) {
  const saltRounds = 12; // যত বেশি, তত ধীর কিন্তু নিরাপদ
  return await bcrypt.hash(plainText, saltRounds);

// ✅ Sensitive data এনক্রিপ্ট করা
const crypto = require('crypto');
const algorithm = 'aes-256-gcm';
const key = Buffer.from(process.env.DB_ENCRYPTION_KEY, 'hex');

function encrypt(text) {
  const iv = crypto.randomBytes(16);
  const cipher = crypto.createCipheriv(algorithm, key, iv);
  let encrypted = cipher.update(text, 'utf8', 'hex');
  encrypted += cipher.final('hex');
  const tag = cipher.getAuthTag();
  return iv.toString('hex') + ':' + tag.toString('hex') + ':' + encrypted;
}

echo "🔄 Backup শুরু: $DATE"
mysqldump -u"$DB_USER" -p"$DB_PASS" \
  --single-transaction --routines --triggers \
  "$DB_NAME" | gzip > "$FILE"

if [ $? -eq 0 ]; then
  echo "✅ Backup সফল: $FILE"

  # AWS S3-এ আপলোড
  aws s3 cp "$FILE" "$S3_BUCKET/"

  # পুরনো Backup মুছুন
  find "$BACKUP_DIR" -name "*.sql.gz" -mtime +"$RETENTION_DAYS" -delete
  echo "🗑️ ${RETENTION_DAYS} দিনের বেশি পুরনো backup মুছে ফেলা হয়েছে"
else
  echo "❌ Backup ব্যর্থ! Alert পাঠাচ্ছি..."
  # Email/Slack alert পাঠান
fi

গুরুত্বপূর্ণ MySQL Metrics

SQL — Monitoring Queries
-- Connection ব্যবহার
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Max_used_connections';

-- Query Performance
SHOW STATUS LIKE 'Slow_queries';
SHOW STATUS LIKE 'Queries';

-- Cache Hit Rate (InnoDB Buffer Pool)
SELECT
  FORMAT(
    (1 - (
      VARIABLE_VALUE / (SELECT VARIABLE_VALUE
        FROM performance_schema.global_status
        WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests')
    )) * 100, 2
  ) AS hit_rate_percent
FROM performance_schema.global_status
WHERE VARIABLE_NAME = 'Innodb_buffer_pool_reads';
-- ৯৫%+ হলে ভালো, কম হলে Buffer Pool বাড়ান

-- এখন চলমান Query দেখুন
SHOW FULL PROCESSLIST;

-- সবচেয়ে ধীর Query খুঁজুন
SELECT
  digest_text,
  count_star AS call_count,
  ROUND(avg_timer_wait/1000000000, 3) AS avg_seconds,
  ROUND(sum_timer_wait/1000000000, 3) AS total_seconds
FROM performance_schema.events_statements_summary_by_digest
ORDER BY sum_timer_wait DESC
LIMIT 10;

-- Table-এর আকার দেখুন
SELECT
  table_name,
  ROUND(data_length/1024/1024, 2) AS data_mb,
  ROUND(index_length/1024/1024, 2) AS index_mb,
  table_rows
FROM information_schema.TABLES
WHERE table_schema = 'ecommerce_db'
ORDER BY data_length DESC;

-- ✅ Online DDL: Table Lock ছাড়া Column যোগ (MySQL 8+)
ALTER TABLE orders
  ADD COLUMN coupon_code VARCHAR(50) DEFAULT NULL,
  ALGORITHM=INPLACE,   -- Table copy করে না
  LOCK=NONE;           -- Read/Write চলতে থাকে

Performance Checklist

✅ করবেন

  • WHERE column-এ Index দিন
  • Prepared Statement ব্যবহার করুন
  • Connection Pool ব্যবহার করুন
  • Slow Query Log চালু রাখুন
  • নিয়মিত ANALYZE TABLE চালান
  • Buffer Pool ৭০% RAM-এ রাখুন
  • Pagination-এ LIMIT/OFFSET ব্যবহার করুন

❌ করবেন না

  • SELECT * ব্যবহার করবেন না
  • Loop-এ Query করবেন না (N+1)
  • Index column-এ Function ব্যবহার করবেন না
  • Root দিয়ে App চালাবেন না
  • Transaction দীর্ঘ করবেন না
  • OFFSET লক্ষাধিক করবেন না
  • Backup ছাড়া Migration করবেন না
