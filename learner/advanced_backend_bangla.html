01
Microservices Architecture

বড় সিস্টেমকে ছোট ছোট সার্ভিসে ভাগ করো

Monolith = একটি বিশাল অ্যাপ। Microservices = প্রতিটি feature আলাদা সার্ভিস, আলাদা deploy, আলাদা scale। Netflix, Amazon, Uber — সবাই এই pattern ব্যবহার করে।

🏗️ Monolith vs Microservices — পার্থক্য
❌ Monolithic Architecture
  • সব কোড একটি codebase-এ
  • একটি জায়গায় bug = পুরো সিস্টেম down
  • একটি ভাষায় সব লিখতে হয়
  • scale করতে হলে পুরোটাই scale করো
  • ছোট টিমে ভালো, বড় হলে nightmare
✅ Microservices Architecture
  • প্রতিটি service আলাদা process
  • একটি service fail = বাকিগুলো চলে
  • প্রতিটি service ভিন্ন ভাষায় লেখা যায়
  • শুধু heavy service-ই scale করো
  • বড় টিমে parallel development সম্ভব
🚪 API Gateway Pattern
Kong / AWS API Gateway

সব client request একটি জায়গায় আসে। Gateway সেই request সঠিক microservice-এ পাঠায়। Authentication, rate limiting, logging সব gateway-তেই।

  • Single entry point — client জানে না ভেতরে কতগুলো service
  • Kong = open-source, plugin-based API Gateway
  • Rate limiting plugin দিয়ে abuse ঠেকাও
  • JWT auth plugin — প্রতি service-এ auth লিখতে হয় না
# Kong-এ route তৈরি করো curl -i -X POST http://localhost:8001/services/ \ --data "name=user-service" \ --data "url=http://user-svc:3000" curl -i -X POST http://localhost:8001/services/user-service/routes \ --data "paths[]=/api/users" # Rate limiting plugin যোগ করো curl -X POST http://localhost:8001/plugins/ \ --data "name=rate-limiting" \ --data "config.minute=100"
🔍 Service Discovery
Services কীভাবে একে অপরকে খোঁজে

Microservices dynamic environment-এ চলে — IP বদলায়, instances বাড়ে-কমে। Service Discovery হলো সেই directory যেখানে সব service নিজেকে register করে।

  • Client-side: client নিজেই registry query করে (Eureka)
  • Server-side: Load Balancer registry query করে (AWS ALB)
  • Kubernetes-এ DNS-based discovery built-in
  • Consul = popular service registry + health check
# Kubernetes-এ Service Discovery # user-service পাবে এই DNS-এ: user-service.default.svc.cluster.local # Environment variable-এও পাওয়া যায় USER_SERVICE_HOST=10.0.0.5 USER_SERVICE_PORT=3000
📡 Inter-Service Communication
Services কীভাবে কথা বলে

দুটো পদ্ধতি: Synchronous (REST/gRPC) এবং Asynchronous (Kafka/RabbitMQ)। কোনটা কখন ব্যবহার করবে সেটা জানাই architecture-এর মূল দক্ষতা।

  • REST: সহজ, HTTP, stateless — CRUD operation-এ
  • gRPC: দ্রুত, binary, type-safe — internal service-এ
  • Kafka: async, event streaming — loosely coupled হতে
  • GraphQL: flexible query, API aggregation-এ
🏛️ Design Patterns
Saga, CQRS, Strangler Fig

Microservices-এ কিছু পরিচিত সমস্যা আছে — distributed transaction, data consistency, migration। এই patterns সেই সমস্যা সমাধান করে।

  • Saga Pattern: distributed transaction → choreography বা orchestration
  • CQRS: Read ও Write model আলাদা করো performance বাড়াতে
  • Circuit Breaker: downstream fail হলে fast-fail করো (Resilience4j)
  • Strangler Fig: পুরনো monolith ধীরে ধীরে microservices-এ migrate করো

02
Apache Kafka

Event Streaming-এর রাজা

LinkedIn বানিয়েছিল, এখন সবাই ব্যবহার করে। প্রতিদিন ট্রিলিয়ন event process করতে পারে। Real-time data pipeline এবং event-driven architecture-এর মূল ভিত্তি।

📋
Topic
Log ফাইলের মতো। Event গুলো Topic-এ লেখা হয়। যেমন: "order-placed", "payment-done", "user-registered"। Producer লেখে, Consumer পড়ে। Immutable — একবার লিখলে বদলানো যায় না।
# Topic তৈরি করো kafka-topics.sh --create \ --topic order-placed \ --partitions 6 \ --replication-factor 3 \ --bootstrap-server localhost:9092
Log-based storage
🗂️
Partition
Topic-কে ভাগ করা হয় Partition-এ। ৬টি Partition = ৬টি Consumer একসাথে পড়তে পারে। Partition key দিয়ে নির্ধারণ হয় কোন message কোন partition-এ যাবে। একই key = সবসময় একই partition → ordering guarantee।
Partition 0
msg 0,3,6...
Partition 1
msg 1,4,7...
Partition 2
msg 2,5,8...
Horizontal scale
📤
Producer
Event পাঠায় Topic-এ। Partition key দিয়ে নির্দিষ্ট partition-এ পাঠাতে পারো। acks=all দিলে সব replica লেখার পর confirm আসে — data loss নেই।
# Python Producer from kafka import KafkaProducer import json producer = KafkaProducer( bootstrap_servers=['localhost:9092'], value_serializer=lambda v: json.dumps(v).encode() ) event = {'order_id': 'ORD-123', 'amount': 500} producer.send('order-placed', key=b'ORD-123', value=event) producer.flush()
Event publisher
📥
Consumer & Consumer Group
Topic থেকে event পড়ে। Consumer Group-এ একাধিক consumer থাকতে পারে। Group-এ ৩টি consumer + ৬ partition = প্রতিটি consumer ২টি partition পায়। Offset দিয়ে কোথায় পড়া শেষ হয়েছে মনে রাখে।
# Python Consumer from kafka import KafkaConsumer consumer = KafkaConsumer( 'order-placed', group_id='payment-service', bootstrap_servers=['localhost:9092'], auto_offset_reset='earliest' ) for msg in consumer: order = json.loads(msg.value) process_payment(order)
Offset tracking
🚀 Kafka কেন এত দ্রুত? (High Throughput-এর রহস্য)
Sequential Write

Kafka disk-এ sequential write করে। Random write-এর চেয়ে sequential write ১০০x দ্রুত। HDD-তেও লক্ষ লক্ষ message/second।

Zero-Copy

OS-এর sendfile() syscall ব্যবহার করে। Data kernel থেকে সরাসরি network-এ যায়, CPU copy লাগে না।

Batching

একসাথে অনেক message batch করে পাঠায়। Network round-trip কমে, throughput বাড়ে।

Compression

lz4, snappy, gzip compression। Network bandwidth কম লাগে, faster transfer।

🎯
Kafka কখন ব্যবহার করবে: ১) High-throughput event streaming (লক্ষ event/sec) ২) Audit log — সব event permanently রাখতে চাই ৩) Multiple consumer — একই event ভিন্ন service process করবে ৪) Event sourcing architecture। কখন করবে না: Simple task queue, low volume, request-reply pattern।
03
RabbitMQ

Message Queue-এর চ্যাম্পিয়ন

AMQP protocol-এর সেরা implementation। Task queue, work distribution, complex routing — এসব কাজে Kafka-র চেয়ে সহজ এবং বেশি উপযুক্ত।

🔀 Exchange
Message Router

Producer সরাসরি Queue-তে পাঠায় না — Exchange-এ পাঠায়। Exchange ঠিক করে message কোন Queue-তে যাবে। চার ধরনের Exchange।

  • Direct: Routing key exact match — email.send → email queue
  • Topic: Wildcard match — *.error → সব error queue-তে
  • Fanout: সব Queue-তে broadcast — notification পাঠাতে
  • Headers: Message header দিয়ে route করো
# Python - pika library import pika, json conn = pika.BlockingConnection( pika.ConnectionParameters('localhost')) ch = conn.channel() # Direct exchange declare ch.exchange_declare( exchange='task_exchange', exchange_type='direct') # Message publish ch.basic_publish( exchange='task_exchange', routing_key='email.send', body=json.dumps({'to': 'user@email.com'}), properties=pika.BasicProperties(delivery_mode=2))
💀 Dead Letter Queue (DLQ)
Failed Message-এর গন্তব্য

Message process করতে fail হলে কোথায় যাবে? Dead Letter Queue-এ। কেন fail হলো সেটা পরে debug করা যায়, message হারায় না।

  • Retry limit পার হলে DLQ-তে যায়
  • Message TTL expire হলে DLQ-তে যায়
  • Queue full হলে DLQ-তে যায়
  • Alert সেট করো: DLQ-তে message জমলে notify হও
# DLQ সহ Queue declare করো ch.queue_declare( queue='email-queue', durable=True, arguments={ 'x-dead-letter-exchange': 'dlx', 'x-dead-letter-routing-key': 'email.failed', 'x-message-ttl': 30000, # 30s TTL 'x-max-retries': 3 })

💡
বাস্তব উদাহরণ: ই-কমার্সে — Order place হলে Kafka-তে event যায় (audit, analytics-এর জন্য replay দরকার)। কিন্তু Email পাঠানো, SMS notification — এগুলো RabbitMQ-তে task দাও। একই system-এ দুটোই থাকতে পারে।
04
System Design

বড় সিস্টেম ডিজাইন করার শিল্প

HLD (High Level Design) থেকে LLD (Low Level Design), CAP Theorem থেকে Sharding — Interview-এ এবং real life-এ সব জায়গায় এই concept লাগে।

⚖️ CAP Theorem — distributed system-এর সবচেয়ে গুরুত্বপূর্ণ নিয়ম

তিনটির মধ্যে যেকোনো দুটো পাবে, তিনটি একসাথে পাওয়া অসম্ভব।

🔒
Consistency (C)

সব node-এ একই data। একটি write হলে সব জায়গায় দেখা যাবে।

⚡
Availability (A)

সবসময় response পাবে। Node down থাকলেও কেউ না কেউ জবাব দেবে।

🌐
Partition Tolerance (P)

Network split হলেও system চলে। Real distributed system-এ P সবসময় থাকতে হবে।

📌 তাই বাস্তবে choice হয় CP বা AP: MySQL, PostgreSQL = CP (consistent, network issue হলে unavailable)। Cassandra, DynamoDB = AP (always available, eventually consistent)।
🗃️ Database Sharding
বিশাল Database ভাগ করো

একটি database-এ ১ বিলিয়ন row — query slow। Sharding মানে data ভাগ করে আলাদা database-এ রাখা।

  • Horizontal Sharding: Row ভাগ করো — User 1-1M → DB1, 1M-2M → DB2
  • Hash Sharding: user_id % 4 = কোন shard-এ যাবে নির্ধারণ হয়
  • Range Sharding: Date বা ID range অনুযায়ী ভাগ
  • Hotspot সমস্যা: একটি shard-এ বেশি load — consistent hashing দিয়ে সমাধান
⚡ Caching Strategy
Redis দিয়ে ১০০x দ্রুত করো

Database hit না করে memory থেকে data দিলে ms-এ response পাবে। Cache কোথায় রাখবে এবং কখন invalidate করবে — সেটাই strategy।

  • Cache-aside: Miss হলে DB থেকে নিয়ে cache-এ রাখো (সবচেয়ে প্রচলিত)
  • Write-through: DB-তে লেখার সাথে সাথে cache update
  • Write-back: আগে cache, পরে DB — দ্রুত কিন্তু data loss risk
  • TTL: নির্দিষ্ট সময় পরে cache expire — stale data ঠেকাও
# Cache-aside in Python def get_user(user_id): cached = redis.get(f"user:{user_id}") if cached: return json.loads(cached) user = db.query("SELECT * FROM users WHERE id=?", user_id) redis.setex(f"user:{user_id}", 3600, json.dumps(user)) return user
🚦 Rate Limiting
API Abuse ঠেকাও

একটি client অনেক বেশি request করলে সার্ভার crash করতে পারে। Rate limiting দিয়ে প্রতিটি client-এর request সীমা নির্ধারণ করো।

  • Token Bucket: Bucket-এ token জমে, request এলে token খরচ হয় — burst allow
  • Sliding Window: গত ১ মিনিটে কতটি request — precise
  • Fixed Window: প্রতি মিনিটে ১০০ request — সহজ কিন্তু edge case আছে
  • Redis দিয়ে distributed rate limiting — সব server এক counter দেখে
🌍 CDN Strategy
Static Content বিশ্বজুড়ে দ্রুত পৌঁছাও

User Bangladesh-এ, server US-এ — latency বেশি। CDN edge server Bangladesh-এর কাছে থাকে, সেখান থেকে serve করে।

  • Images, JS, CSS, Video — সব CDN-এ দাও
  • Cache-Control header: max-age=31536000 (1 year)
  • Cache busting: filename-এ hash যোগ করো (main.abc123.js)
  • Origin Shield: CDN miss হলে origin hit কমাও
🏛️ HLD vs LLD — দুই স্তরের Design
High Level Design (HLD)

Architecture level — boxes and arrows। Interview-এ প্রথমে এটা।

  • কোন components থাকবে? (Services, DB, Cache, CDN)
  • Components কীভাবে communicate করবে?
  • Data flow কেমন হবে?
  • Scalability কীভাবে হবে?
Low Level Design (LLD)

Code level — class, method, schema। Interview-এ দ্বিতীয় পর্যায়।

  • Database schema কেমন হবে?
  • API endpoints কী কী?
  • Class diagram, design patterns
  • Algorithms এবং data structures
🎯
System Design Interview Formula: ১) Requirements clarify করো ২) Capacity estimation করো (QPS, storage) ৩) HLD আঁকো ৪) Deep dive করো specific component-এ ৫) Bottleneck identify করো ও solve করো। "Design URL shortener" জিজ্ঞেস করলে — শুরুতেই জিজ্ঞেস করো "কতটি URL/day? Read-heavy নাকি write-heavy?"

GitOps / ArgoCD

Git-ই সত্যের উৎস

Infrastructure এবং application config সব Git-এ থাকে। Git change হলে automatically cluster-এ apply হয়। Manual kubectl apply করার দিন শেষ।

📚
GitOps কী?
Git repository = Single Source of Truth। Kubernetes manifest, Helm chart, Kustomize — সব Git-এ। কেউ cluster-এ manually change করলে GitOps সেটা revert করে দেয়। "Desired state" সবসময় Git-এ লেখা থাকে।
  • Pull-based deployment — cluster নিজেই Git দেখে change নেয়
  • সব change Git commit → audit trail, rollback সহজ
  • Developer শুধু Git-এ push করে, বাকিটা অটোমেটিক
Declarative
🔄
ArgoCD — GitOps Engine
Kubernetes-এর জন্য সবচেয়ে popular GitOps tool। Git repo দেখে, cluster-এর current state দেখে, পার্থক্য থাকলে sync করে। Beautiful UI আছে — কোন app কোথায়, কী status সব দেখা যায়।
# ArgoCD install করো kubectl create namespace argocd kubectl apply -n argocd -f \ https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml # Application তৈরি করো argocd app create my-app \ --repo https://github.com/myorg/my-app-config \ --path k8s/overlays/production \ --dest-server https://kubernetes.default.svc \ --dest-namespace production \ --sync-policy automated \ --auto-prune \ --self-heal

Service Mesh — Istio

Microservices-এর মধ্যে নিরাপদ যোগাযোগ

Microservices বাড়লে inter-service communication জটিল হয়। Istio এই জটিলতা handle করে — code একটুও না বদলে। Sidecar proxy pattern ব্যবহার করে।

🔀 Traffic Management
কোথায় কতটুকু Traffic যাবে

Canary deploy, Blue-Green deploy, Traffic mirroring — code না লিখে YAML দিয়ে করো। নতুন version-এ ১০% traffic পাঠাও, সব ঠিক থাকলে ১০০%।

# Canary deployment — 10% নতুন version-এ apiVersion: networking.istio.io/v1alpha3 kind: VirtualService spec: hosts: ["user-service"] http: - route: - destination: host: user-service subset: v1 weight: 90 - destination: host: user-service subset: v2 weight: 10
🔐 mTLS (Mutual TLS)
Service-to-Service Encryption

সব service-এর মধ্যে encrypted connection। Service A কে নিশ্চিত হতে হবে সে Service B-র সাথে কথা বলছে, অন্য কারো সাথে না। Zero Trust Network।

  • Istio automatically certificate generate করে
  • Certificate rotation অটোমেটিক — মেয়াদ শেষ হলে নতুন হয়
  • STRICT mode — শুধু mTLS connection allow
  • অ্যাপ কোড কিছুই জানে না — sidecar handle করে
⚡ Circuit Breaking
Cascade Failure ঠেকাও

Service B slow হলে Service A-ও slow হয়, chain reaction-এ সব down হয়। Circuit Breaker এই cascade failure ঠেকায়।

  • Closed: সব ঠিক, request যায়
  • Open: বেশি fail হলে circuit break — fast fail
  • Half-Open: কিছু সময় পর test request পাঠায়
  • Fallback response দাও — graceful degradation
# Circuit Breaker config trafficPolicy: outlierDetection: consecutiveErrors: 5 interval: 10s baseEjectionTime: 30s maxEjectionPercent: 100
📊 Observability
সব দেখো, সব জানো

Istio automatic metrics, traces, logs generate করে। কোন service slow, কোথায় error, request কোন path দিয়ে গেছে — সব Kiali dashboard-এ দেখো।

  • Prometheus metrics — Grafana-তে visualize
  • Jaeger/Zipkin — Distributed tracing
  • Kiali — Service graph, traffic visualization
  • Access log — প্রতিটি request-এর record
⚠️
Istio-র Overhead জানো: প্রতিটি pod-এ Envoy sidecar inject হয় — CPU ও Memory বেশি লাগে। Small cluster বা simple microservices-এ Istio না নিয়ে শুধু Kubernetes NetworkPolicy + NGINX Ingress যথেষ্ট। Istio নাও যখন ৫০+ microservice, complex traffic management, বা mTLS সত্যিই দরকার।

Testing Strategy

Test না করলে Production-এ কান্না

Unit থেকে E2E, Load testing থেকে TDD — একটি সম্পূর্ণ testing strategy ছাড়া confident deployment সম্ভব না। Testing Pyramid জানো।

🔺 Testing Pyramid — কত % কোন ধরনের test
E2E Tests
~10% — ধীর, ব্যয়বহুল
Integration Tests
~20% — component মিলিয়ে test
Unit Tests
~70% — দ্রুত, সস্তা, বেশি রাখো
🔬 Unit Testing
একটি function, একটি test

একটি function বা method আলাদাভাবে test করো। External dependency mock করো। দ্রুত চলে, CI-তে সব সময় চালাও।

# Jest (Node.js) Unit Test describe('calculateTotal', () => { it('should apply discount correctly', () => { const result = calculateTotal(1000, 10); expect(result).toBe(900); }); it('should throw on negative price', () => { expect(() => calculateTotal(-100, 0)) .toThrow('Invalid price'); }); });
🔗 Integration Testing
Database, API — মিলিয়ে test করো

Service layer + Database একসাথে। Real database ব্যবহার করো (TestContainers দিয়ে Docker-এ)। API endpoint test করো।

# Supertest — API Integration Test const request = require('supertest'); describe('POST /api/users', () => { it('creates user and returns 201', async () => { const res = await request(app) .post('/api/users') .send({ name: 'রফিক', email: 'rafiq@bd.com' }); expect(res.statusCode).toBe(201); expect(res.body.name).toBe('রফিক'); }); });
🎭 E2E — Playwright / Cypress
Browser দিয়ে পুরো flow test

Real browser চালাও, user-এর মতো click করো, form fill করো, result check করো। Playwright সবচেয়ে modern এবং দ্রুত।

# Playwright E2E Test import { test, expect } from '@playwright/test'; test('user can login and see dashboard', async ({ page }) => { await page.goto('https://myapp.com/login'); await page.fill('#email', 'user@test.com'); await page.fill('#password', 'pass123'); await page.click('button[type=submit]'); await expect(page.locator('h1')) .toContainText('Dashboard'); });
🚀 Load Testing — k6 / Locust
হাজার user একসাথে সামলাতে পারবে?

Production যাওয়ার আগে জানো — ১০০০ concurrent user হলে API কত ms response দেয়? কোথায় bottleneck? কত load-এ crash করে?

// k6 Load Test Script import http from 'k6/http'; import { check, sleep } from 'k6'; export const options = { stages: [ { duration: '2m', target: 100 }, // ramp up { duration: '5m', target: 500 }, // hold { duration: '2m', target: 0 }, // ramp down ], thresholds: { http_req_duration: ['p(95)<500'] } }; export default function () { const res = http.get('https://api.myapp.com/products'); check(res, { 'status 200': r => r.status === 200 }); sleep(1); }
🔴🟢🔵 TDD — Test Driven Development

আগে test লেখো, তারপর code। Red-Green-Refactor cycle।

🔴
RED

প্রথমে failing test লেখো। Feature এখনো নেই, তাই fail হবেই।

🟢
GREEN

Minimum code লেখো যাতে test pass হয়। Perfect code না।

🔵
REFACTOR

Code clean করো। Test green থাকলে refactor নিরাপদ।

✅
Testing Coverage লক্ষ্য: ১০০% coverage অর্থহীন। Critical business logic-এ ৯০%+, API layer-এ ৮০%+, UI component-এ ৭০%+ রাখলেই ভালো। Coverage metric-এর চেয়ে গুরুত্বপূর্ণ হলো সঠিক জায়গায় সঠিক test। একটি ভালো integration test দশটি খারাপ unit test-এর চেয়ে মূল্যবান।
08
Platform Engineering

Developer-দের জন্য সোনার রাস্তা

DevOps বিবর্তিত হয়ে হয়েছে Platform Engineering। Developer-দের self-service দাও — infrastructure নিজে provision করুক, pipeline নিজে তৈরি করুক।

🏗️ Internal Developer Platform (IDP)
Developer-দের পোর্টাল

Developer নিজেই নতুন service তৈরি করতে পারে, deploy করতে পারে, log দেখতে পারে — Ops team-এর টিকিট না কেটে। Backstage (Spotify) সবচেয়ে popular IDP framework।

  • Service Catalog — সব service একজায়গায় discover করো
  • Software Templates — নতুন service template থেকে তৈরি
  • TechDocs — documentation সার্ভিসের পাশে
  • Plugin system — GitHub, PagerDuty, Kubernetes integrate
🛤️ Golden Path
সেরা পথটা সহজ করো

Security, compliance, best practice সব মেনে service তৈরির "approved way"। Developer এই path follow করলে সব ঠিক থাকে। Custom করার freedom আছে, কিন্তু golden path-ই সহজ।

  • Cookiecutter / Yeoman template — service boilerplate
  • Pre-configured CI/CD pipeline — commit করলেই deploy
  • Observability built-in — metrics, logs, traces by default
  • Security scanning — image vulnerability, SAST অটোমেটিক
# Backstage template উদাহরণ apiVersion: scaffolder.backstage.io/v1beta3 kind: Template metadata: name: nodejs-microservice spec: steps: - fetch:plain # template download - publish:github # repo তৈরি - catalog:register # IDP-তে register
🤖 Self-Service Infrastructure
Infra Provisioning অটোমেটিক

Developer নতুন database দরকার? Ops-কে বলো না — form fill করো, Terraform automatically create হয়। Crossplane বা Terraform-এর Service Catalog এই কাজ করে।

  • Crossplane — Kubernetes CRD দিয়ে cloud resource manage
  • Terraform modules — approved, tested infra building blocks
  • Policy as Code (OPA) — কে কী provision করতে পারবে
  • Cost visibility — কার resource কত টাকা লাগছে
📏 Platform Metrics
Platform-এর সাফল্য কীভাবে মাপবে

Platform team-এর সফলতা মাপা হয় Developer Productivity দিয়ে। DORA Metrics এর জন্য industry standard।

  • Deployment Frequency: কতবার/দিনে deploy হয়?
  • Lead Time for Changes: commit থেকে production কত সময়?
  • MTTR: incident হলে recover করতে কত সময়?
  • Change Failure Rate: কত % deploy problem সৃষ্টি করে?
