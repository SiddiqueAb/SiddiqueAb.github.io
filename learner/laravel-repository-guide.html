01

এই প্যাটার্ন কী এবং কেন ব্যবহার করি?

Repository Pattern মানে হলো প্রতিটি কাজের জন্য আলাদা জায়গা। Controller database query করে না, Service validation করে না, Repository business decision নেয় না।

সহজ উদাহরণ: একটা রেস্তোরাঁতে waiter (Controller) order নেয়, chef (Service) রান্না করার সিদ্ধান্ত নেয়, আর pantry (Repository) উপকরণ সরবরাহ করে। Waiter নিজে রান্না করে না।

এই প্যাটার্নের ফায়দা
যেকোনো layer আলাদাভাবে test করা যায়। MySQL বাদ দিয়ে MongoDB দিলে শুধু Repository বদলাতে হবে, Controller বা Service ছুঁতে হবে না। Team-এর যেকেউ যেকোনো layer-এ কাজ করতে পারবে conflicts ছাড়া।

02

৭টি Layer — কে কী করে

IF
Interface — চুক্তিপত্র
Repository-কে কী কী method implement করতে হবে সেটা define করে। ভবিষ্যতে DB বদলালে এই file ছুঁতে হবে না।
কখনো business logic লেখা যাবে না
RE
Repository — Database Access
শুধু Eloquent query। create(), find(), update(), delete()। "Customer active কিনা" এই সিদ্ধান্ত এখানে নয়।
কোনো if/business rule লেখা যাবে না
SV
Service — Business Logic
সব business সিদ্ধান্ত এখানে। "Customer বানালে email পাঠাতে হবে?" — এখানে decide হয়। Repository-কে call করে।
কখনো HTTP request/response দেখবে না
CO
Controller — HTTP Layer
Request নাও, Service-কে দাও, Resource দিয়ে response পাঠাও। মাত্র ৩টা কাজ। DB query কখনো না।
Customer::create() লেখা মানেই ভুল
FR
FormRequest — Validation
সব validation rule এখানে। Controller-এ $request->validate() লেখা যাবে না। আলাদা class বানাতেই হবে।
শুধু rules() আর authorize() থাকবে
RS
API Resource — JSON Transform
Model-এর data কীভাবে JSON হবে সেটা এখানে। Client-কে কোন field দেখাবে, কোনটা লুকাবে।
Response-এ কখনো $model->toArray() দেবে না
PR
Service Provider — DI Binding
Laravel-কে বলে "Interface চাইলে কোন Class দাও।" একবার লিখলেই সব জায়গায় কাজ করে।
প্রতিটি Interface bind করতে হবে
03

Interface — চুক্তিপত্র

Interface হলো একটা চুক্তি। এই file-এ বলা থাকে — "যে class-ই এই কাজ করুক, তাকে অবশ্যই এই method গুলো implement করতে হবে।" এর ফলে Service জানে না Repository MySQL ব্যবহার করে নাকি MongoDB।

কেন Interface লাগবে? ধরো ভবিষ্যতে MySQL-এর বদলে MongoDB ব্যবহার করবে। তখন শুধু MongoCustomerRepository বানিয়ে Provider-এ bind করো। Service বা Controller-এর একটা লাইনও বদলাতে হবে না।
app/Repositories/Interfaces/CustomerRepositoryInterface.php 
<?php

namespace App\Repositories\Interfaces;

use Illuminate\Contracts\Pagination\LengthAwarePaginator;
use Illuminate\Database\Eloquent\Model;

interface CustomerRepositoryInterface
{
    // সব Repository-কে এই method গুলো implement করতে হবে
    public function paginate(int $perPage = 15): LengthAwarePaginator;
    public function findById(int $id): ?Model;
    public function findOrFail(int $id): Model;
    public function create(array $data): Model;
    public function update(int $id, array $data): Model;
    public function delete(int $id): bool;
    public function findByEmail(string $email): ?Model;
}
04

Repository — শুধু Database

Repository-তে শুধু Eloquent query থাকবে। এখানে কোনো business logic নেই। "Customer delete করার আগে invoice check করতে হবে" — এই কথা এখানে লেখা যাবে না, এটা Service-এর কাজ।

app/Repositories/CustomerRepository.php 
<?php

namespace App\Repositories;

use App\Models\Customer;
use App\Repositories\Interfaces\CustomerRepositoryInterface;
use Illuminate\Contracts\Pagination\LengthAwarePaginator;
use Illuminate\Database\Eloquent\Model;

class CustomerRepository implements CustomerRepositoryInterface
{
    public function __construct(
        private readonly Customer $model
    ) {}

    public function paginate(int $perPage = 15): LengthAwarePaginator
    {
        // শুধু query — কোনো condition নেই
        return $this->model->latest()->paginate($perPage);
    }

    public function findOrFail(int $id): Model
    {
        return $this->model->findOrFail($id);
    }

    public function create(array $data): Model
    {
        return $this->model->create($data);
    }

    public function update(int $id, array $data): Model
    {
        $customer = $this->findOrFail($id);
        $customer->update($data);
        return $customer->fresh();
    }

    public function delete(int $id): bool
    {
        return $this->findOrFail($id)->delete();
    }

    public function findByEmail(string $email): ?Model
    {
        return $this->model->where('email', $email)->first();
    }
}
05

Service — Business Logic-এর বাড়ি

Service হলো সবচেয়ে গুরুত্বপূর্ণ layer। এখানে সব business সিদ্ধান্ত থাকে। Customer বানানো হলে email পাঠাতে হবে? Delete করার আগে invoice check করতে হবে? এসব এখানে।

গুরুত্বপূর্ণ: Service কখনো $request দেখে না। Service শুধু plain array বা value নেয়। এর মানে Service-কে API ছাড়া অন্য জায়গা থেকেও (যেমন CLI command) call করা যাবে।
app/Services/CustomerService.php 
<?php

namespace App\Services;

use App\Repositories\Interfaces\CustomerRepositoryInterface;
use Illuminate\Contracts\Pagination\LengthAwarePaginator;
use Illuminate\Database\Eloquent\Model;

class CustomerService
{
    public function __construct(
        private readonly CustomerRepositoryInterface $repo
    ) {}

    public function paginate(int $perPage = 15): LengthAwarePaginator
    {
        return $this->repo->paginate($perPage);
    }

    public function create(array $data): Model
    {
        $customer = $this->repo->create($data);

        // Business Logic: customer তৈরি হলে event fire করো
        // event(new CustomerCreated($customer));

        // Business Logic: welcome email পাঠাও
        // Mail::to($customer->email)->send(new WelcomeMail($customer));

        return $customer;
    }

    public function findOrFail(int $id): Model
    {
        return $this->repo->findOrFail($id);
    }

    public function update(int $id, array $data): Model
    {
        return $this->repo->update($id, $data);
    }

    public function delete(int $id): void
    {
        // Business Rule: unpaid invoice থাকলে delete করা যাবে না
        $customer = $this->repo->findOrFail($id);

        // if ($customer->invoices()->where('status','unpaid')->exists()) {
        //     throw new \DomainException('Unpaid invoices exist.');
        // }

        $this->repo->delete($id);
    }
}
06

Controller — মাত্র ৩টা কাজ

Controller-এর সর্বোচ্চ ৩টা কাজ: ১) Request নাও, ২) Service-কে দাও, ৩) Resource দিয়ে response পাঠাও। এর বাইরে Controller কিছু করবে না।

✓ করবে

FormRequest inject করবে। Service method call করবে। Resource দিয়ে response return করবে।

✗ করবে না

Customer::create() লিখবে না। $request->validate() লিখবে না। if দিয়ে business rule লিখবে না।

app/Http/Controllers/Api/V1/CustomerController.php 
<?php

namespace App\Http\Controllers\Api\V1;

use App\Http\Controllers\Controller;
use App\Http\Requests\V1\StoreCustomerRequest;
use App\Http\Requests\V1\UpdateCustomerRequest;
use App\Http\Resources\V1\CustomerResource;
use App\Services\CustomerService;
use Illuminate\Http\JsonResponse;
use Illuminate\Http\Resources\Json\AnonymousResourceCollection;

class CustomerController extends Controller
{
    public function __construct(
        private readonly CustomerService $customerService
    ) {}

    // ১) Request নাও → ২) Service call করো → ৩) Resource return করো

    public function index(): AnonymousResourceCollection
    {
        return CustomerResource::collection(
            $this->customerService->paginate()
        );
    }

    public function store(StoreCustomerRequest $request): JsonResponse
    {
        $customer = $this->customerService->create($request->validated());

        return response()->json(new CustomerResource($customer), 201);
    }

    public function show(int $id): JsonResponse
    {
        return response()->json(
            new CustomerResource($this->customerService->findOrFail($id))
        );
    }

    public function update(UpdateCustomerRequest $request, int $id): JsonResponse
    {
        $customer = $this->customerService->update($id, $request->validated());

        return response()->json(new CustomerResource($customer));
    }

    public function destroy(int $id): JsonResponse
    {
        $this->customerService->delete($id);

        return response()->json(null, 204);
    }
}
07

FormRequest — Validation-এর বাড়ি

সব validation এখানে। Controller-এ $request->validate() লেখা এই প্যাটার্নে নিষেধ। আলাদা Request class বানাতে হবে। V1 আর V2-এর জন্য আলাদা Request folder।

app/Http/Requests/V1/StoreCustomerRequest.php 
<?php

namespace App\Http\Requests\V1;

use Illuminate\Foundation\Http\FormRequest;

class StoreCustomerRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true; // Policy ব্যবহার করলে এখানে check করো
    }

    public function rules(): array
    {
        return [
            'name'    => ['required', 'string', 'max:255'],
            'email'   => ['required', 'email', 'unique:customers,email'],
            'phone'   => ['nullable', 'string', 'max:20'],
            'address' => ['nullable', 'string'],
        ];
    }

    public function messages(): array
    {
        return [
            'email.unique' => 'এই email দিয়ে আগেই customer আছে।',
        ];
    }
}

API Resource — JSON Transform

Resource ঠিক করে Client কোন data দেখবে। Model-এর সব field expose করা নিরাপদ নয়। password, remember_token এগুলো Resource দিয়ে লুকানো যায়। V1 আর V2-এর Resource আলাদা — এটাই versioning-এর মূল কৌশল।

app/Http/Resources/V1/CustomerResource.php 
<?php

namespace App\Http\Resources\V1;

use Illuminate\Http\Request;
use Illuminate\Http\Resources\Json\JsonResource;

class CustomerResource extends JsonResource
{
    public function toArray(Request $request): array
    {
        return [
            'id'         => $this->id,
            'name'       => $this->name,
            'email'      => $this->email,
            'phone'      => $this->phone,
            'address'    => $this->address,
            'created_at' => $this->created_at?->toDateTimeString(),
            // V2-তে হয়তো 'invoices_count' যোগ হবে — V1 অপরিবর্তিত থাকবে
        ];
    }
}

Service Provider — DI Binding

Laravel-কে বলতে হবে — "কেউ CustomerRepositoryInterface চাইলে CustomerRepository দাও।" এটা না করলে Laravel জানবে না কোন class inject করতে হবে এবং error আসবে।

app/Providers/RepositoryServiceProvider.php 
<?php

namespace App\Providers;

use App\Repositories\CustomerRepository;
use App\Repositories\Interfaces\CustomerRepositoryInterface;
use App\Repositories\InvoiceRepository;
use App\Repositories\Interfaces\InvoiceRepositoryInterface;
use Illuminate\Support\ServiceProvider;

class RepositoryServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        // নতুন Repository যোগ করলে এখানে bind করতে হবে
        $this->app->bind(
            CustomerRepositoryInterface::class,
            CustomerRepository::class
        );

        $this->app->bind(
            InvoiceRepositoryInterface::class,
            InvoiceRepository::class
        );
    }
}

এরপর bootstrap/app.php-এ register করো:

bootstrap/app.php 
return Application::configure(basePath: dirname(__DIR__))
    ->withProviders([
        \App\Providers\RepositoryServiceProvider::class,
    ])
    // ... বাকি config

নতুন Feature যোগ করার নিয়ম

যেকোনো নতুন feature যোগ করার সময় এই ৮টা ধাপ follow করবে। একটাও skip করবে না। উদাহরণ: Invoice feature যোগ করছি।

Artisan দিয়ে files তৈরি করো
terminal 
php artisan make:model Invoice -m
php artisan make:controller Api/V1/InvoiceController --api

V2 তৈরির নিয়ম — কখন ও কীভাবে

V2 মানে পুরো project আবার লেখা নয়। শুধু যা বদলাবে তা নতুন করে লিখতে হবে। Service আর Repository একই থাকবে — সেগুলো V1 আর V2 দুটোই ব্যবহার করবে।

কখন V2 লাগবে?

পরিবর্তনের ধরনV2 লাগবে?উদাহরণ
নতুন field response-এ যোগনাcreated_at যোগ করা
নতুন endpoint যোগনাGET /customers/{id}/orders
Field rename করাহ্যাঁphone → phone_number
Field type বদলানোহ্যাঁprice: float → string
Field সরিয়ে দেওয়াহ্যাঁaddress বাদ দেওয়া
Response structure পরিবর্তনহ্যাঁdata wrapper সরানো

V2 বানানোর ধাপ

ধরো V2-তে Customer response-এ invoices_count আর last_purchase যোগ হবে এবং phone-এর নাম বদলে mobile হবে।

V1 — অপরিবর্তিত থাকে
✓ routes/api_v1.php
✓ Controllers/Api/V1/CustomerController
✓ Requests/V1/StoreCustomerRequest
✓ Resources/V1/CustomerResource

এই files ছুঁবে না।
Production-এ এখনো চলছে।
V2 — নতুন তৈরি করো
+ routes/api_v2.php
+ Controllers/Api/V2/CustomerController
+ Requests/V2/StoreCustomerRequest
+ Resources/V2/CustomerResource

Service ও Repository একই।
Business logic duplicate নয়।
routes/api.php — V2 যোগ করার পর 
// V1 — production-এ live, কখনো ভাঙবে না
Route::prefix('v1')->group(base_path('routes/api_v1.php'));

// V2 — নতুন features, breaking changes allowed
Route::prefix('v2')->group(base_path('routes/api_v2.php'));
app/Http/Resources/V2/CustomerResource.php — V2-তে নতুন fields 
class CustomerResource extends JsonResource
{
    public function toArray(Request $request): array
    {
        return [
            'id'             => $this->id,
            'name'           => $this->name,
            'email'          => $this->email,
            'mobile'         => $this->phone, // V2: renamed from phone
            'invoices_count' => $this->invoices_count, // V2: new field
            'last_purchase'  => $this->last_purchase, // V2: new field
        ];
    }
}
মূল নিয়ম: V2-তে CustomerService আর CustomerRepository একটুও বদলাবে না। শুধু Controller, Resource, আর Request নতুন বানাবে। Business logic কখনো duplicate করবে না।

করণীয় ও বর্জনীয় — Quick Reference

Controller-এ

✓ করবে
public function store(StoreCustomerRequest $r)
{
    $c = $this->service->create($r->validated());
    return new CustomerResource($c);
}
✗ করবে না
public function store(Request $r)
{
    $r->validate(['name'=>'required']);
    Customer::create($r->all()); // ❌
}

Service-এ

✓ করবে
public function create(array $data): Model
{
    $c = $this->repo->create($data);
    event(new CustomerCreated($c));
    return $c;
}
✗ করবে না
public function create(Request $r) // ❌ Request!
{
    Customer::create($r->all()); // ❌ Direct query!
}

Repository-তে

✓ করবে
public function delete(int $id): bool
{
    // শুধু delete — check নেই
    return $this->findOrFail($id)->delete();
}
✗ করবে না
public function delete(int $id): bool
{
    // ❌ Business rule Repository-তে নয়!
    if ($this->hasUnpaidInvoices($id)) {
        throw new \Exception('...');
    }
}

VS Code Agent Prompt

এই prompt-টা তোমার VS Code-এর AI Agent-কে দাও। যেকোনো নতুন feature বানাতে বললে Agent এই pattern follow করবে।

ব্যবহার: VS Code-এ Copilot Chat বা যেকোনো AI extension খুলো। এই prompt copy করে দাও। তারপর বলো "InvoiceController বানাও" বা "Product feature যোগ করো" — Agent এই pattern অনুসরণ করবে।
// VS CODE AGENT SYSTEM PROMPT — ERP LARAVEL PROJECT
You are a Laravel 12 API developer working on an ERP project. You MUST strictly follow the Repository Pattern architecture described below. Never deviate from this structure. ## PROJECT STACK - Laravel 12 + Laravel Sanctum - Repository Pattern with Service Layer - PSR-4 autoloading, PSR-12 coding standard - API versioning: /api/v1, /api/v2 ## MANDATORY FOLDER STRUCTURE When creating any new feature (e.g. "Invoice"), create files in this exact order: 1. app/Repositories/Interfaces/{Name}RepositoryInterface.php 2. app/Repositories/{Name}Repository.php 3. app/Services/{Name}Service.php 4. app/Http/Requests/V1/Store{Name}Request.php 5. app/Http/Requests/V1/Update{Name}Request.php 6. app/Http/Resources/V1/{Name}Resource.php 7. app/Http/Controllers/Api/V1/{Name}Controller.php 8. Add binding in app/Providers/RepositoryServiceProvider.php 9. Add routes in routes/api_v1.php ## LAYER RULES — NEVER BREAK THESE ### Interface - Only method signatures, no logic - Return types must be declared - Use Model, LengthAwarePaginator, bool etc. from Illuminate contracts ### Repository - Implements the Interface - Constructor receives readonly Model via DI - ONLY Eloquent queries — no if statements, no business logic - Methods: paginate, findById, findOrFail, create, update, delete - update() must call fresh() before returning ### Service - Constructor receives readonly Interface (not concrete class) via DI - ALL business logic lives here — events, emails, validation rules beyond form - NEVER receives Request object — only plain array or scalar values - NEVER calls Eloquent directly — always via Repository ### Controller - Constructor receives readonly Service via DI - ONLY 3 jobs: receive request → call service → return resource - NEVER call Eloquent directly (no Customer::create()) - NEVER call $request->validate() — use FormRequest - NEVER write business logic or if conditions - HTTP status codes: 200 GET/PUT, 201 POST, 204 DELETE, 401 unauth, 422 validation ### FormRequest - authorize() returns true (unless using Policies) - rules() returns validation array - messages() for custom error messages in Bangla if needed - Version-specific: V1 requests in Requests/V1/, V2 in Requests/V2/ ### API Resource - toArray() returns only fields the client should see - Never expose: password, remember_token, internal flags - V1 and V2 have SEPARATE resource files — never share - Use optional() for nullable relations ### Service Provider (RepositoryServiceProvider) - Every new Interface MUST be bound to its concrete class here - Use $this->app->bind(Interface::class, Concrete::class) - Register in bootstrap/app.php withProviders() ## API VERSIONING RULES When V2 is needed (breaking changes only): - Create: routes/api_v2.php, Controllers/Api/V2/, Resources/V2/, Requests/V2/ - NEVER create new Service or Repository for V2 — reuse V1 Services - NEVER duplicate business logic between versions - V1 files are FROZEN — never modify them after production deployment Breaking changes that REQUIRE a new version: - Renaming a response field - Changing a field's data type - Removing a field from response - Changing authentication mechanism Non-breaking changes (no version bump needed): - Adding new optional response fields - Adding new endpoints - Adding new optional request parameters ## CODE STYLE (PSR-12) - Constructor property promotion with readonly: private readonly ServiceClass $service - Return type declarations on ALL methods - No inline validation ($request->validate()) - use statements alphabetically sorted - Method names: camelCase, Classes: PascalCase - Strict typing preferred ## ARTISAN COMMANDS TO RUN php artisan make:controller Api/V1/{Name}Controller --api php artisan make:request V1/Store{Name}Request php artisan make:request V1/Update{Name}Request php artisan make:resource V1/{Name}Resource php artisan make:model {Name} -m Then manually create: Interface, Repository, Service ## VERIFICATION After creating any feature, run: php artisan route:list --path=api php artisan test --filter={Name} If asked to create a feature, always output ALL files in the order listed above. Never skip the Interface. Never bind directly to a concrete class in the Service constructor.

Cursor / Windsurf-এর জন্য (.cursorrules file)

Project root-এ .cursorrules নামে একটা file বানাও এবং উপরের prompt সেখানে paste করো। তাহলে প্রতিবার prompt না দিলেও Agent এই rules follow করবে।

terminal 
# Project root-এ .cursorrules file তৈরি করো
touch .cursorrules
# উপরের prompt paste করো

# Git-এ track করো যাতে team-ও পায়
git add .cursorrules
git commit -m "chore: add cursor rules for repository pattern"
Pro Tip: VS Code GitHub Copilot Chat-এ কাজ করলে .github/copilot-instructions.md file-এ এই prompt রাখো। Copilot automatically এটা context হিসেবে নেবে। Cursor IDE-তে .cursorrules সবচেয়ে ভালো কাজ করে।
