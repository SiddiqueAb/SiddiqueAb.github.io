DevOps কী?

Development এবং Operations — এই দুই টিমের মধ্যে সেতুবন্ধন তৈরি করাই DevOps এর মূল লক্ষ্য

আগে যখন কোনো software তৈরি হতো, Developer টিম কোড লিখত এবং Operations টিম সেই কোড server-এ deploy করত। কিন্তু এই দুই টিমের মধ্যে সবসময় ঝামেলা লেগে থাকত।

Developer বলত: "আমার computer-এ তো ঠিকঠাক চলছে!" আর Operations বলত: "Server-এ দিলেই crash করছে!" এই সমস্যার সমাধান হলো DevOps.

🎯 সহজ ভাষায়
DevOps মানে হলো — Code লেখা, Test করা, Build করা, Deploy করা এবং Monitor করা — এই পুরো প্রক্রিয়াকে দ্রুত, নির্ভরযোগ্য এবং স্বয়ংক্রিয় করা।

DevOps এর তিনটি মূল স্তম্ভ:

  • Culture: Dev ও Ops টিম একসাথে কাজ করবে, blame করবে না
  • Automation: Repetitive কাজ machine দিয়ে করাও
  • Measurement: সব কিছু measure করো — speed, errors, uptime
  • Sharing: Knowledge ও responsibility শেয়ার করো
❌ DevOps ছাড়া
  • Deploy করতে ২ সপ্তাহ লাগে
  • Manual testing, অনেক ভুল
  • Production crash হলে কে দায়ী বোঝা যায় না
  • Dev ও Ops এর মধ্যে দোষারোপ
✅ DevOps সহ
  • প্রতিদিন বা প্রতিঘণ্টায় deploy
  • Automated testing, কম ভুল
  • সবাই সব কিছুর দায়িত্ব নেয়
  • দ্রুত সমস্যা সমাধান
⚡
Speed (গতি)
Amazon প্রতি ১১ সেকেন্ডে একবার deploy করে। DevOps ছাড়া এটা সম্ভব না।
🛡️
Reliability (নির্ভরযোগ্যতা)
Automated test ও rollback থাকায় production-এ সমস্যা হলে তাৎক্ষণিক ফিরে আসা যায়।
📏
Scale (বড় হওয়া)
১০০ user থেকে ১ কোটি user — infrastructure automatically scale হয়।
🔒
Security (নিরাপত্তা)
DevSecOps: Security শুরু থেকেই pipeline-এ যুক্ত থাকে।

CI/CD Pipeline

Continuous Integration এবং Continuous Deployment — DevOps এর হৃদয়

CI — Continuous Integration কী?

প্রতিবার কেউ কোড push করলে automatically সেই কোড test হয় এবং build হয়। ধরো ৫ জন developer একসাথে কাজ করছে — CI নিশ্চিত করে তাদের কোড একসাথে ভালোভাবে কাজ করছে।

CD — Continuous Deployment কী?

CI সফল হলে automatically production-এ deploy হয়। কোনো manual কাজ নেই। Code push করো → Test pass → Deploy! 🎉

💡 উদাহরণ দিয়ে বুঝি
তুমি একটি e-commerce site এর developer। নতুন "ছাড়" feature add করলে। Git-এ push করার ২০ মিনিটের মধ্যে সেই feature লাখো user দেখতে পাচ্ছে — কোনো manual কাজ ছাড়াই!

⚡ GitHub Actions — সবচেয়ে জনপ্রিয় CI/CD Tool

এই YAML ফাইলটি .github/workflows/deploy.yml এ রাখলেই CI/CD শুরু হয়ে যায়:

YAML — .github/workflows/deploy.yml
# GitHub Actions CI/CD Pipeline — বাংলা ই-কমার্স প্রজেক্টের জন্য

name: CI/CD Pipeline

# কখন এই pipeline চলবে?
on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  # ধাপ ১: Test করা
  test:
    runs-on: ubuntu-latest
    steps:
      - name: কোড নামিয়ে আনা
        uses: actions/checkout@v3
      
      - name: Node.js সেটআপ
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      
      - name: Dependencies install করা
        run: npm install
      
      - name: Unit test চালানো
        run: npm test
      
      - name: Code quality check
        run: npm run lint

  # ধাপ ২: Docker Image তৈরি ও Push
  build-and-push:
    needs: test  # test পাস না হলে এটা চলবে না
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Docker Hub-এ login করা
        uses: docker/login-action@v2
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}
      
      - name: Docker image তৈরি ও push
        run: |
          docker build -t myapp:${{ github.sha }} .
          docker push myapp:${{ github.sha }}

  # ধাপ ৩: Production-এ Deploy
  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # শুধু main branch এ
    steps:
      - name: Server-এ deploy করা
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.SERVER_IP }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_KEY }}
          script: |
            docker pull myapp:${{ github.sha }}
            docker stop myapp-container || true
            docker run -d --name myapp-container -p 80:3000 myapp:${{ github.sha }}
            echo "✅ Deploy সফল হয়েছে!"

🐳 Docker — Container Technology

"আমার computer-এ চলে" এই সমস্যার সমাধান!

Docker কেন দরকার?

ধরো তুমি একটি রান্না করলে বাড়িতে — সুন্দর হলো। কিন্তু রেস্তোরাঁয় গিয়ে একই রান্না করলে ভিন্ন স্বাদ! কারণ উপকরণ ও পরিবেশ আলাদা।

Docker হলো রান্নার সম্পূর্ণ Recipe + উপকরণ + রান্নাঘর একসাথে প্যাক করার মতো। যেখানেই নিয়ে যাও, একই ফলাফল।

🐳 Docker এর চার মূল ধারণা
  • Image: রান্নার Recipe — একটি template
  • Container: চলমান application — Recipe থেকে রান্না
  • Dockerfile: Image তৈরির নির্দেশনা
  • Registry: Image store করার জায়গা (Docker Hub)

# Node.js Application এর Dockerfile

# ধাপ ১: Base Image — Node.js 18 Alpine (হালকা version)
FROM node:18-alpine

# ধাপ ২: Container-এর ভেতরে কাজের ফোল্ডার
WORKDIR /app

# ধাপ ৩: package.json আগে copy করা (cache এর জন্য)
COPY package*.json ./

# ধাপ ৪: Dependencies install করা
RUN npm ci --only=production

# ধাপ ৫: সব কোড copy করা
COPY . .

# ধাপ ৬: কোন Port ব্যবহার হবে
EXPOSE 3000

# ধাপ ৭: Non-root user (নিরাপত্তার জন্য)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# ধাপ ৮: Container চালু হলে কী করবে
CMD ["node", "server.js"]

☸️ Kubernetes — Container Orchestration

হাজারো Container পরিচালনা করার ব্যবস্থাপক

🤔 কেন Kubernetes দরকার?
Docker দিয়ে একটি container চালাতে পারো। কিন্তু যদি ১০০০ container চালাতে হয়? কোনটা crash করলে restart করবে কে? Traffic বাড়লে আপনা-আপনি আরও container চালু হবে কীভাবে? — এই সব সমস্যার সমাধান Kubernetes (K8s)।

Kubernetes এর মূল Concepts:

🏘️ Cluster
অনেকগুলো server মিলে একটি cluster। একটি Master node বাকিদের নিয়ন্ত্রণ করে।
📦 Pod
K8s এ সবচেয়ে ছোট unit। এক বা একাধিক container নিয়ে একটি Pod।
🔄 Deployment
কতটি Pod চলবে, কীভাবে update হবে তা নির্ধারণ করে।
🌐 Service
Pod গুলোতে traffic পাঠায়, load balancing করে।
deployment.yaml
# K8s Deployment — 3টি replica চলবে
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3  # ৩টি copy চলবে
  selector:
    matchLabels:
      app: myapp
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v1.0
        ports:
        - containerPort: 3000
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:  # Crash হলে restart
          httpGet:
            path: /health
            port: 3000
---
# Service — বাইরে থেকে access দেওয়া
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 3000
  selector:
    app: myapp

📊 Monitoring & Observability

Production-এ কী হচ্ছে না জানলে কীভাবে সমস্যা সমাধান করবে?

তিনটি স্তম্ভ (Three Pillars):

Pillar 1
📈 Metrics
CPU, Memory, Request rate, Error rate এর সংখ্যাগত তথ্য। Prometheus collect করে, Grafana সুন্দর graph বানায়।
Pillar 2
📝 Logs
Application কী করছে তার বিস্তারিত লেখা। Error হলে কোথায় হলো বোঝা যায়। ELK Stack (Elasticsearch, Logstash, Kibana) জনপ্রিয়।
Pillar 3
🔍 Traces
একটি request কোন কোন service এর মধ্য দিয়ে গেছে তার পথ। Jaeger বা Zipkin ব্যবহার হয়।

📋 Step-by-Step: Developer থেকে Production পর্যন্ত

  1. Developer কোড লেখে ও push করে

    নতুন "Flash Sale" feature branch তৈরি করে কোড লিখলো। git push origin feature/flash-sale করলো।

  2. Pull Request ও Code Review

    GitHub-এ PR তৈরি করলো। Team lead code review করলো। Automated linting ও security scan চললো।

  3. CI Pipeline চালু হলো

    PR merge হতেই GitHub Actions চালু হলো। ২৫০টি unit test চললো, সব pass করলো। Docker image তৈরি হলো।

  4. Staging Environment-এ Deploy

    Production-এর মতো একটি test environment-এ আগে deploy হলো। QA team manual testing করলো।

  5. Production Deployment — Blue-Green Strategy

    পুরনো version (Blue) চলতে থাকলো। নতুন version (Green) deploy হলো। ধীরে ধীরে ১০% → ৫০% → ১০০% traffic Green-এ গেলো।

  6. Health Check ও Monitoring

    Grafana dashboard দেখলো — error rate ০.০১%, response time স্বাভাবিক। সব ঠিক আছে! Blue environment বন্ধ করা হলো।

🏗️ Terraform দিয়ে AWS Infrastructure

Code দিয়ে server তৈরি! Manual console-এ click করার দরকার নেই:

main.tf — AWS Infrastructure
# Terraform দিয়ে AWS Infrastructure তৈরি

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
  # State file S3 এ রাখা — team এর সবাই একই state দেখবে
  backend "s3" {
    bucket = "banglashop-terraform-state"
    key    = "prod/terraform.tfstate"
    region = "ap-southeast-1"
  }
}

provider "aws" {
  region = "ap-southeast-1"  # Singapore region
}

# EKS Cluster (Kubernetes)
module "eks" {
  source          = "terraform-aws-modules/eks/aws"
  cluster_name    = "banglashop-prod"
  cluster_version = "1.29"
  
  node_groups = {
    general = {
      desired_size = 3  # ৩টি server
      min_size     = 2  # কমপক্ষে ২টি
      max_size     = 10 # সর্বোচ্চ ১০টি
      instance_types = ["t3.medium"]
    }
  }
}

# RDS PostgreSQL Database
resource "aws_db_instance" "main" {
  identifier        = "banglashop-db"
  engine            = "postgres"
  engine_version    = "15.4"
  instance_class    = "db.t3.medium"
  allocated_storage = 100  # 100 GB
  
  multi_az          = true   # High availability
  backup_retention_period = 7  # ৭ দিনের backup
  
  tags = {
    Environment = "production"
    Project     = "BanglaShop"
  }
}

# Redis Cache
resource "aws_elasticache_cluster" "main" {
  cluster_id   = "banglashop-cache"
  engine       = "redis"
  node_type    = "cache.t3.micro"
  num_cache_nodes = 1
}
💡 Terraform ব্যবহারের সুবিধা
এই ফাইল থেকে পুরো AWS infrastructure তৈরি হবে। terraform init → terraform plan → terraform apply — মাত্র ৩টি command! এবং সব কিছু Git-এ থাকবে তাই ইতিহাস জানা যাবে।

🚨 Production Incident — কীভাবে সামলাবে?

⚠️ Scenario: রাত ২টায় website down!
Grafana-তে alert এলো — Error rate ৪৫%, Response time ৩০ সেকেন্ড। কী করবে?
Incident Response Steps
# ধাপ ১: কী হচ্ছে বোঝো
kubectl get pods -n production
# Output: myapp-xxx   0/1   CrashLoopBackOff   ← সমস্যা এখানে!

kubectl logs myapp-xxx --previous
# Output: Error: Cannot connect to database

# ধাপ ২: সাথে সাথে rollback করো (আগের version-এ ফিরে যাও)
kubectl rollout undo deployment/myapp
# ৩০ সেকেন্ডেই আগের stable version চলবে!

kubectl rollout status deployment/myapp
# Waiting for rollout to finish...
# deployment "myapp" successfully rolled out ✅

# ধাপ ৩: কারণ খোঁজো
kubectl describe pod myapp-xxx
# Events: database connection pool exhausted

# ধাপ ৪: Fix করো ও আবার deploy করো (সকালে)
# database connection limit বাড়ানো হলো
# নতুন version deploy হলো
# Postmortem লেখা হলো — ভবিষ্যতে এটা যাতে না হয়
