01📋

পরিচিতি ও নৈতিক নীতিমালা

Bug Bounty কি?

Bug Bounty হলো এমন একটি প্রোগ্রাম যেখানে কোম্পানি বা সংস্থা তাদের সিস্টেমের নিরাপত্তা দুর্বলতা খুঁজে বের করার জন্য ethical হacker বা security researcher-দের পুরস্কার দেয়। এটি একটি আইনি, কাঠামোবদ্ধ পদ্ধতি যেখানে কোম্পানি নির্দিষ্ট সীমার মধ্যে তাদের সিস্টেম পরীক্ষা করার অনুমতি দেয়।

🎯
Ethical Hacker
অনুমতি নিয়ে সিস্টেম পরীক্ষা করে দুর্বলতা খোঁজেন এবং রিপোর্ট করেন

📚
Security Researcher
নতুন vulnerability ও attack প্যাটার্ন গবেষণা করেন

নৈতিক নীতিমালা — এটাই সবচেয়ে গুরুত্বপূর্ণ

⚠️ আইনি সতর্কতা

অনুমতি ছাড়া কোনো সিস্টেম পরীক্ষা করা সম্পূর্ণ অবৈধ। বাংলাদেশের Digital Security Act 2018 এবং আন্তর্জাতিক Computer Fraud and Abuse Act (CFAA) অনুযায়ী এটি গুরুতর অপরাধ। শুধুমাত্র নিজের সিস্টেম, অনুমোদিত lab (HackTheBox, TryHackMe), অথবা Bug Bounty scope-এর মধ্যে পরীক্ষা করুন।

✅ Ethical Hacker-এর নিয়ম

১. সবসময় লিখিত অনুমতি নিন।  ২. Bug Bounty-এর scope মেনে চলুন।  ৩. পাওয়া তথ্য গোপন রাখুন।  ৪. Production data কখনো access করবেন না।  ৫. খুঁজে পাওয়া vulnerability দ্রুত রিপোর্ট করুন।

Ethical Hacking vs Malicious Hacking

আইনি অবস্থাসম্পূর্ণ আইনিঅবৈধ — জেল হতে পারে
ডেটাকিছু নেওয়া হয় নাচুরি বা ধ্বংস

Reconnaissance & Footprinting

Passive Reconnaissance — কোনো সরাসরি যোগাযোগ ছাড়া

Passive recon-এ target-এর server-এ সরাসরি কোনো request না পাঠিয়ে publicly available তথ্য সংগ্রহ করা হয়।

Passive Recon Sources
Google Dorking: site:example.com filetype:pdf → Public PDF files site:example.com inurl:admin → Admin pages site:example.com ext:php inurl:id= → PHP param endpoints site:example.com "index of" → Directory listing "example.com" filetype:env → Exposed .env files Shodan (shodan.io): org:"Example Company" → All their IPs/services hostname:example.com → Open ports & services ssl:"example.com" → SSL certificates Certificate Transparency (crt.sh): https://crt.sh/?q=%.example.com → All issued certificates → Subdomain discovery without scanning GitHub Code Search: org:examplecompany password → Leaked credentials org:examplecompany api_key → Exposed API keys filename:.env "DB_PASSWORD" → Config files

Technology Fingerprinting

🔍 Recon Commands (Terminal)
# Nmap — Port & Service Discovery
# শুধুমাত্র অনুমোদিত target-এ ব্যবহার করুন
nmap -sV -sC -p 80,443,8080,8443 example.com
# -sV: service version detection
# -sC: default scripts
# Output দেখায়: Apache 2.4.41, PHP/7.4, MySQL 8.0

# WhatWeb — Technology Detection
whatweb https://example.com
# Output: WordPress[5.8], jQuery[3.6], PHP[7.4], Apache

# Wappalyzer browser extension
# Browser install করুন — site visit করলেই সব tech দেখাবে

# HTTP Headers analysis
curl -I https://example.com
# দেখুন: Server, X-Powered-By, Set-Cookie, Security headers

# JavaScript file enumeration
# Browser DevTools → Sources → JS files-এ endpoint খোঁজুন
# গুরুত্বপূর্ণ: /api/v1/users, /admin/dashboard

# Wayback Machine — পুরনো URL
# web.archive.org/web/*/example.com/*
# পুরনো endpoint, parameter, admin panel খোঁজুন

# Nikto — Web Server Scanner
nikto -h https://example.com
# Common misconfigs, outdated software, insecure headers
⚠️ Active Scanning নোট

Nmap, Nikto-র মতো active scanning tools সরাসরি target-এর server-এ request পাঠায়। শুধুমাত্র আপনার নিজের সিস্টেম বা Bug Bounty scope-এর মধ্যে ব্যবহার করুন। Unauthorized scanning অবৈধ।

Essential Tools & Resources

Burp Suite — Web Application Testing-এর প্রধান হাতিয়ার

Burp Suite হলো ethical hacker-দের সবচেয়ে বেশি ব্যবহৃত web security testing tool। এটি আপনার browser এবং target website-এর মধ্যে একটি proxy হিসেবে কাজ করে।

Burp Suite Architecture
Browser ──── HTTP Request ────► Burp Proxy (127.0.0.1:8080) ────► Target Server Browser ◄─── HTTP Response ──── Burp Proxy ◄────────────────────── Target Server Burp Suite Main Modules: ┌─────────────┬───────────────────────────────────────────────────┐ │ Proxy │ সব request/response intercept ও modify করো │ │ Repeater │ একটি request বারবার modify করে পাঠাও │ │ Intruder │ Automated fuzzing — parameter testing │ │ Scanner │ Automatic vulnerability scan (Pro version) │ │ Decoder │ Base64, URL, HTML encode/decode │ │ Comparer │ দুটি response-এর পার্থক্য দেখো │ │ Logger │ সব HTTP traffic log করো │ └─────────────┴───────────────────────────────────────────────────┘

Burp Suite Setup (বাংলায়)

  1. 1
    Download করুন: portswigger.net/burp/communitydownload — Community Edition বিনামূল্যে
  2. 2
    Browser-এ proxy: 127.0.0.1:8080 set করুন (Firefox recommended, FoxyProxy extension ব্যবহার করুন)
  3. 3
    Burp CA certificate install করুন: http://burp → CA Certificate download → Browser-এ import
  4. 4
    Proxy → Intercept ON করুন → Browser-এ target site browse করুন → সব request দেখতে পাবেন
  5. 5
    Request-এ right-click → "Send to Repeater" → modify করে পাঠান, response দেখুন

05💉

SQL Injection (SQLi)

SQL Injection কী?

SQL Injection হলো OWASP Top 10-এর সবচেয়ে বিপজ্জনক vulnerability। যখন user input সরাসরি SQL query-তে যুক্ত করা হয় — কোনো sanitization ছাড়া — তখন attacker SQL code inject করে database-এর সম্পূর্ণ নিয়ন্ত্রণ নিতে পারে।

Critical Database access Data theft Authentication bypass Admin access

কিভাবে দুর্বলতা তৈরি হয়

❌ Vulnerable PHP Code
// User input সরাসরি query-তে
$username = $_POST['username'];
$password = $_POST['password'];

$sql = "SELECT * FROM users
  WHERE username='$username'
  AND password='$password'";

$result = mysqli_query($conn, $sql);

// Input: admin' -- -
// Query হয়:
// SELECT * FROM users
// WHERE username='admin'-- -'
// AND password='...'
// password check skip হয়!
✅ Secure PHP Code
// Prepared Statement ব্যবহার করুন
$username = $_POST['username'];
$password = $_POST['password'];

$stmt = $conn->prepare(
  "SELECT * FROM users
   WHERE username = ?
   AND password_hash = ?"
);
$stmt->bind_param("ss",
  $username, hash('sha256',$password)
);
$stmt->execute();
// User input এখন data হিসেবে
// treat হয়, SQL হিসেবে নয়

SQL Injection Types

Typeকিভাবে কাজ করেউদাহরণ
In-band (Error)Error message-এ data দেখায়Database version, table names
In-band (Union)UNION দিয়ে অন্য table-এর dataUser credentials চুরি
Blind (Boolean)True/False response-এ data বের করাAdmin account আছে কিনা
Blind (Time-based)Response time দিয়ে data বের করাSLEEP() function abuse
Out-of-bandDNS/HTTP callback দিয়ে dataAir-gapped systems

SQLi Test করার পদ্ধতি (Authorized Lab-এ)

💉 SQLi Detection Techniques
-- Step 1: Basic test — একটি single quote দিন
-- URL: http://lab.local/item?id=1'
-- SQL Error দেখালে → potentially vulnerable

-- Step 2: Boolean test
-- URL: http://lab.local/item?id=1 AND 1=1  → normal response
-- URL: http://lab.local/item?id=1 AND 1=2  → different/empty
-- Different response → blind SQLi সম্ভব

-- Step 3: Comment test
-- MySQL:  ' --  বা  ' #
-- MSSQL:  ' --
-- Oracle: ' --

-- Step 4: UNION column count
-- ORDER BY 1-- → ORDER BY 2-- → error না হওয়া পর্যন্ত
-- Error হলে আগেরটাই column count

-- DVWA/WebGoat Lab-এ উদাহরণ:
-- Login form-এ username field-এ: admin'-- -
-- এটি password check skip করে login দেয় (vulnerable app-এ)

-- SQLMap — authorized target এ automated test
-- sqlmap -u "http://lab.local/item?id=1" --dbs
-- শুধুমাত্র নিজের DVWA/WebGoat lab-এ ব্যবহার করুন

Prevention — Secure Coding

✅ PHP — Prepared Statements (PDO)
// ✅ PDO Prepared Statement — সবচেয়ে নিরাপদ পদ্ধতি
try {
    $pdo = new PDO('mysql:host=localhost;dbname=mydb', $user, $pass, [
        PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
        PDO::ATTR_EMULATE_PREPARES => false  // Real prepared statements
    ]);

    // ✅ Named parameter
    $stmt = $pdo->prepare(
        "SELECT id, name, email FROM users
         WHERE username = :username AND active = 1"
    );
    $stmt->execute([':username' => $_POST['username']]);
    $user = $stmt->fetch(PDO::FETCH_ASSOC);

} catch (PDOException $e) {
    // Error details user-কে দেখাবেন না!
    error_log($e->getMessage());
    die('Database error occurred');
}

// ✅ Input validation additional layer
function sanitizeInput($input, $type = 'string') {
    switch($type) {
        case 'int':
            return filter_var($input, FILTER_VALIDATE_INT);
        case 'email':
            return filter_var($input, FILTER_VALIDATE_EMAIL);
        default:
            return htmlspecialchars(strip_tags(trim($input)));
    }
}
06📜

Cross-Site Scripting (XSS)

XSS কী এবং কতটা বিপজ্জনক?

XSS হলো এমন একটি vulnerability যেখানে attacker একটি website-এ malicious JavaScript inject করতে পারে। সেই script অন্য user-এর browser-এ execute হয়। এটি দিয়ে session চুরি, keylogging, phishing করা সম্ভব।

High Cookie theft Session hijacking Defacement Credential phishing

XSS-এর তিনটি প্রকার

Typeকিভাবে কাজ করেকোথায় হয়Persistence
Reflected XSSURL parameter-এ script, server reflect করেSearch forms, error messagesNon-persistent
Stored XSSDatabase-এ script save হয়, সবাই দেখলে executeComments, profile, messagesPersistent — বেশি বিপজ্জনক
DOM-based XSSClient-side JS-এ URL data processSingle-page appsNon-persistent
❌ Vulnerable Code (PHP)
// User input সরাসরি HTML-এ
$name = $_GET['name'];

// Vulnerable output
echo "Hello, " . $name;

// Attacker URL:
// ?name=<script>alert(1)</script>
// ?name=<img src=x onerror=alert(1)>
// ?name=<svg onload=alert(1)>

// Stored XSS in comment:
echo $comment['body'];
// Database থেকে সরাসরি print
✅ Secure Code
// htmlspecialchars — সবসময় output encode
$name = htmlspecialchars(
    $_GET['name'],
    ENT_QUOTES | ENT_HTML5,
    'UTF-8'
);
echo "Hello, " . $name;

// Twig template (auto-escape)
// {{ name }} → auto escaped
// {{ name|raw }} → only if safe

// Content Security Policy header
// header("Content-Security-Policy:
//   default-src 'self';
//   script-src 'self';
//   object-src 'none'");

XSS Testing কীভাবে করবেন (Lab-এ)

📜 XSS Detection — DVWA Lab
-- XSS চেনার basic test (lab environment-এ)
-- Input field-এ test করার জন্য harmless payload:

<script>alert('XSS Test')</script>

-- 2. img tag (script tag block হলে)
<img src="x" onerror="alert('XSS')">

-- 3. Filter bypass test
<svg/onload=alert(1)>

// DOMPurify library ব্যবহার করুন
// const clean = DOMPurify.sanitize(dirtyHTML);
// document.getElementById('output').innerHTML = clean;

Content Security Policy (CSP) — সর্বোত্তম প্রতিরক্ষা

07🔄

Cross-Site Request Forgery (CSRF)

CSRF কিভাবে কাজ করে?

CSRF-এ একটি malicious website ব্যবহার করে একজন authenticated user-এর browser থেকে target website-এ unauthorized request পাঠানো হয়। User জানেও না যে তার account দিয়ে কিছু হয়েছে।

CSRF Attack Flow
১. User bank.com-এ login করে (session cookie আছে) ২. User malicious.com visit করে ③. malicious.com-এ hidden HTML form: <form action="https://bank.com/transfer" method="POST"> <input type="hidden" name="to" value="attacker_account"> <input type="hidden" name="amount" value="50000"> </form> <script>document.forms[0].submit();</script> ৪. Browser auto-submit করে bank.com-এ request পাঠায় ৫. bank.com user-এর valid session cookie দেখে request execute করে ৬. টাকা transfer হয়ে যায় — user কিছু বোঝেনি!
❌ No CSRF Protection
// Form — no CSRF token
<form method="POST"
      action="/transfer">
  <input name="to" .../>
  <input name="amount" .../>
  <button>Transfer</button>
</form>
// যেকোনো site থেকে
// submit করা যায়!
✅ CSRF Token Protected
// PHP — CSRF Token Generate
if (empty($_SESSION['csrf'])) {
  $_SESSION['csrf'] =
    bin2hex(random_bytes(32));
}
// Form-এ hidden field
echo '<input type="hidden"
  name="csrf_token"
  value="'.$_SESSION['csrf'].'">';
// Verify on POST
if ($_POST['csrf_token'] !==
    $_SESSION['csrf']) {
  die('Invalid CSRF token');
}
✅ CSRF Prevention Checklist

① প্রতিটি state-changing request-এ CSRF token। ② SameSite=Strict cookie attribute। ③ Custom request header (API-র জন্য)। ④ Referer/Origin header validation। ⑤ Sensitive action-এ re-authentication।

IDOR — Insecure Direct Object Reference

Bug Bounty-তে সবচেয়ে বেশি পাওয়া vulnerability। একজন user অন্য user-এর resource access করতে পারে।

🔑 IDOR Vulnerability Example
-- ❌ Vulnerable: User ID URL-এ, authorization check নেই
-- URL: /api/profile?user_id=1001
-- Attacker changes to: /api/profile?user_id=1002
-- অন্য user-এর profile দেখা যায়!

-- ❌ Vulnerable PHP code:
$userId = $_GET['user_id'];  // user-supplied!
$profile = getProfile($userId);  // no ownership check
echo json_encode($profile);

-- ✅ Secure: Session-এর user ID ব্যবহার করুন
$userId = $_SESSION['user_id'];  // server-side session
$profile = getProfile($userId);
echo json_encode($profile);

-- ✅ Secure API: ownership verification
$stmt = $pdo->prepare(
  "SELECT * FROM orders
   WHERE id = :order_id
   AND user_id = :session_user_id"
);
$stmt->execute([
  ':order_id' => $_GET['id'],
  ':session_user_id' => $_SESSION['user_id']
]);
-- user_id match না হলে empty result

-- IDOR Testing Methodology:
-- ১. Account A দিয়ে resource তৈরি করুন (note: ID)
-- ২. Account B দিয়ে login করুন
-- ③. Account A-এর ID দিয়ে request করুন
-- ৪. Access পেলে → IDOR vulnerability!

Password Security Best Practices

✅ Secure Password Handling (PHP)
// ✅ Password hashing — bcrypt (PHP built-in)
$hash = password_hash($password, PASSWORD_BCRYPT, ['cost' => 12]);
// cost=12: 2^12 iterations — brute force কঠিন

// ✅ Password verify
if (password_verify($_POST['password'], $storedHash)) {
    // login success
}

// ✅ Rate limiting — brute force prevent
$attempts = $redis->get("login_attempts:" . $ip);
if ($attempts >= 5) {
    die('Too many attempts. Try after 15 minutes.');
}

// ✅ Session regeneration after login
session_regenerate_id(true);  // prevent session fixation

// ✅ JWT Token best practices
// - Strong secret key (256+ bits)
// - Short expiry (15 min access, 7 day refresh)
// - Algorithm: RS256 or HS256 (never 'none'!)
// - Validate 'alg' header explicitly

File Upload Vulnerabilities

File Upload — কেন বিপজ্জনক?

Insecure file upload feature দিয়ে attacker malicious file (webshell) upload করে server-এ code execute করতে পারে। এটি সরাসরি RCE (Remote Code Execution) ঘটাতে পারে।

❌ Vulnerable Upload Code
// Only extension check — bypassable
$ext = pathinfo($_FILES['f']['name'],
              PATHINFO_EXTENSION);
if ($ext == 'php') {
  die('Not allowed');
}
// Bypass: file.php.jpg
// Bypass: file.pHp
// Bypass: file.php%00.jpg
// Bypass: file.php5, file.phtml

// Web-accessible folder-এ save
move_uploaded_file(
  $_FILES['f']['tmp_name'],
  'uploads/' . $_FILES['f']['name']
);
✅ Secure Upload Code
// MIME type check (magic bytes)
$finfo = new finfo(FILEINFO_MIME_TYPE);
$mime = $finfo->file(
  $_FILES['f']['tmp_name']
);
$allowed = ['image/jpeg','image/png'];
if (!in_array($mime, $allowed)) {
  die('Invalid file type');
}
// Random filename
$name = bin2hex(random_bytes(16));
// Web root-এর বাইরে save
// /var/uploads/ (not /var/www/)
// PHP execution বন্ধ করুন:
// .htaccess: php_flag engine off
// Serve via readfile() proxy
📁 Secure File Upload Checklist

① MIME type magic bytes দিয়ে validate। ② Allowed whitelist (jpg, png, pdf only)। ③ Random UUID filename। ④ Web root-এর বাইরে store। ⑤ PHP execution disabled in upload dir। ⑥ Maximum file size limit। ⑦ Antivirus scan (ClamAV)। ⑧ CDN/S3 থেকে serve — same domain-এ না।

Session Management & Cookie Security

Session Hijacking কিভাবে হয়?

Session hijacking-এ attacker একজন user-এর valid session ID চুরি করে তার পরিচয়ে কাজ করে। XSS দিয়ে cookie চুরি, network sniffing, বা session fixation এর মাধ্যমে এটি হতে পারে।

✅ Secure Session Configuration (PHP)
// php.ini বা runtime config
ini_set('session.cookie_httponly', 1);     // JS থেকে access করতে পারবে না
ini_set('session.cookie_secure', 1);       // HTTPS only
ini_set('session.cookie_samesite', 'Strict'); // CSRF prevent
ini_set('session.use_strict_mode', 1);     // Session fixation prevent
ini_set('session.gc_maxlifetime', 1800);   // 30 min timeout
ini_set('session.use_only_cookies', 1);    // URL-এ session ID না

session_start();

// Login-এর পর session regenerate করুন
session_regenerate_id(true);  // পুরনো session delete করে নতুন দেয়

// User fingerprint store করুন
$_SESSION['ip'] = $_SERVER['REMOTE_ADDR'];
$_SESSION['ua'] = md5($_SERVER['HTTP_USER_AGENT']);

// প্রতিটি request-এ verify করুন
if ($_SESSION['ip'] !== $_SERVER['REMOTE_ADDR'] ||
    $_SESSION['ua'] !== md5($_SERVER['HTTP_USER_AGENT'])) {
    session_destroy();
    header('Location: /login');
    exit;
}

// Set-Cookie header example:
// Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict; Path=/
🍪 Cookie Security Attributes

HttpOnly: JavaScript দিয়ে cookie read করা যাবে না (XSS prevent)। Secure: HTTPS-এ only পাঠানো হবে। SameSite=Strict: Cross-site request-এ cookie পাঠানো হবে না (CSRF prevent)। Path ও Domain: Scope সীমিত করুন।

Code Injection & RCE Prevention

Injection Vulnerabilities — বিভিন্ন ধরন

RCE (Remote Code Execution) হলো সবচেয়ে critical vulnerability — attacker সরাসরি server-এ command execute করতে পারে। এটি সাধারণত code injection, deserialization, বা vulnerable library-র মাধ্যমে হয়।

❌ Command Injection (Vulnerable)
// User input সরাসরি shell command-এ
$ip = $_GET['ip'];
// Input: 8.8.8.8; cat /etc/passwd
$output = shell_exec("ping -c 1 " . $ip);
echo $output;
// Server-এ arbitrary command চালাতে পারে!

// eval() দিয়ে PHP injection
eval($_GET['code']);  // Never!

// Unsafe deserialization
$obj = unserialize($_COOKIE['data']);
✅ Safe Code
// escapeshellarg() ব্যবহার করুন
$ip = $_GET['ip'];

// Whitelist validation
if (!filter_var($ip, FILTER_VALIDATE_IP)) {
  die('Invalid IP');
}
$safeIp = escapeshellarg($ip);
$output = shell_exec("ping -c 1 " . $safeIp);

// Better: avoid shell entirely
// Use PHP native functions

// JSON instead of serialize
$data = json_decode($_COOKIE['data'], true);
// Never unserialize user input
⚡ Never Use in Production

PHP: eval(), exec(), shell_exec(), system(), passthru(), unserialize() — user input দিয়ে কখনো না। JavaScript: eval(), innerHTML (unsanitized), document.write()। Template: সরাসরি variable without escaping।

✅ Nginx Security Configuration
# /etc/nginx/conf.d/security.conf
server {
    listen 443 ssl http2;

    # Hide server version
    server_tokens off;

    # Security headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; object-src 'none'; frame-ancestors 'none';" always;

    # Block common attack patterns
    location ~* \.(env|git|htaccess|htpasswd|sql|bak|backup)$ {
        deny all;
    }

    # Rate limiting
    limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;
    location /login {
        limit_req zone=login burst=5 nodelay;
    }
}
