ডেটাটাইপ ক্যালকুলেশন — প্রোর মতো ভাবো

একটাই নিয়ম মনে রাখো: সবচেয়ে ছোট datatype ব্যবহার করো যেটা কখনো overflow হবে না। PRAN-RFL-এ ১০ বাইট বাঁচানো মানে ৫০ কোটি রেকর্ডে = ৫ GB বাঁচানো।

Integer Types — সংখ্যার জন্য সবচেয়ে বেশি ব্যবহৃত
Type বাইট UNSIGNED সর্বোচ্চ কোথায় ব্যবহার করবে প্রো টিপ
TINYINT 1 255 status, is_active, bu_id, rating (১-৫) PRAN-এর সব division_id এটাই
SMALLINT 2 65,535 category_id, warehouse_id, district_id বাংলাদেশে ৬৪ জেলা → SMALLINT
MEDIUMINT 3 ১৬.৭ মিলিয়ন product_id (যদি SKU কম থাকে) INT-এর চেয়ে ১ বাইট ছোট
INT 4 ৪.২৯ বিলিয়ন customer_id, product_id, sales_rep_id Default choice for IDs
BIGINT 8 ১৮ কুইন্টিলিয়ন order_id, transaction_id (daily millions) শুধু সত্যিই দরকার হলে
⚠️
UNSIGNED সবসময় যোগ করো যদি নেগেটিভ কখনো হবে না। INT UNSIGNED দিলে ৪.২৯ বিলিয়ন পাও, INT দিলে মাত্র ২.১৪ বিলিয়ন — একই বাইটে দ্বিগুণ!
String Types — টেক্সটের জন্য সবচেয়ে বেশি ভুল হয় এখানে
Type স্টোরেজ কখন ব্যবহার করবে কখন করবে না
CHAR(n) Fixed n bytes সবসময় same length: BD mobile (11), ISO country code (2), PRAN product code CHAR(10) variable length data-তে — waste হয়
VARCHAR(n) 1-2 + actual chars নাম, ইমেইল, address — সবচেয়ে বেশি ব্যবহৃত INDEX column-এ VARCHAR(500) — ধীর হয়
TINYTEXT 255 bytes max Short notes, tags INDEX করতে হলে ব্যবহার করো না
TEXT Off-page (pointer) Long descriptions, product details, HTML content INDEX করা যায় না — ORDER BY ধীর
ENUM 1-2 bytes মাত্র! Fixed list: status('PENDING','PAID'), type('FOOD','BVRG') নতুন value যোগ করতে ALTER লাগে — সাবধান
🚨
সবচেয়ে বড় ভুল: সব জায়গায় VARCHAR(255) দেওয়া। email সর্বোচ্চ ৮০ char হয়। ব্যবহার করো VARCHAR(80)। ১০ কোটি রেকর্ডে ১৭৫ byte বাঁচবে = ~১.৭ GB!

Money & Decimal — সবচেয়ে গুরুত্বপূর্ণ
Typeসমস্যা
FLOAT / DOUBLE ৳১০০.৫০ → ৳১০০.৪৯৯৯৯৯ হতে পারে! কখনো টাকার জন্য নয়!
DECIMAL(10,2) Exact। price, total, tax সবকিছুতে এটাই। ধীর কিন্তু নির্ভরযোগ্য।
BIGINT (paisa) ৳১০.৫০ = ১০৫০ পয়সা store করো। সবচেয়ে দ্রুত। PRAN-এর পছন্দ।

Index কীভাবে কাজ করে — B+Tree গভীরে

১ কোটি রেকর্ড থেকে একটা row খুঁজতে index ছাড়া MySQL ১ কোটিবার দেখে। Index থাকলে মাত্র ~২৩ বার। এটাই পার্থক্য।

Root Node (মাথা)
1 – 10,000,000
সব key-এর রেঞ্জ
↓   লগারিদমিক অনুসন্ধান

1 – 3.3M
3.3M – 6.6M
6.6M – 10M
↓   id=5,000,000 খুঁজছি → মধ্যের branch

3.3M–4M
4M–4.6M
4.6M–5.3M ✓
5.3M–6M
6M–6.6M
↓   Leaf → Actual Row Data

১ কোটি
Full scan ছাড়া
→
~২৩ বার
B+Tree দিয়ে (log₂ 10M)
🔑
Primary Key (Clustered Index)
InnoDB-তে actual row data PRIMARY KEY-এর B+Tree-তে store হয়। মানে PK দিয়ে খোঁজা সবচেয়ে দ্রুত — data সরাসরি পাওয়া যায়। সবসময় AUTO_INCREMENT INT/BIGINT রাখো।
📌
Secondary Index (Non-clustered)
Secondary index-এ secondary column value + primary key store হয়। তাই ২ বার lookup লাগে: প্রথমে secondary index, তারপর primary index থেকে row। এটাকে বলে "double lookup"।
🔗
Composite Index
KEY idx_bu_date (bu_id, date) — এটা (bu_id) এবং (bu_id, date) দুটো query-ই cover করে। কিন্তু শুধু (date) query-তে কাজ করে না। Left prefix rule মনে রাখো।
🎯
Covering Index
যদি query-র সব column index-এ থাকে, MySQL data file-এই যায় না। শুধু index পড়েই answer দেয়। এটাই সবচেয়ে দ্রুত। PRAN-এর report query-তে এটা critical।
📝
UNIQUE Index
Duplicate prevent করে + automatically index তৈরি করে। SKU, barcode, email, phone — এগুলোতে UNIQUE KEY দাও। Performance bonus পাবে।
⚡
FULLTEXT Index
Product name বা description-এ search করতে। WHERE name LIKE '%juice%' index কাজ করে না — FULLTEXT index দাও। PRAN-এর product search-এ দরকার।
⚠️
Index-এর ফাঁদ: বেশি index = INSERT/UPDATE ধীর হয়। প্রতি write-এ সব index update হয়। PRAN-এর order_items table-এ ৫-৬টার বেশি index রেখো না। শুধু যে column-এ WHERE/JOIN/ORDER BY লাগে সেখানে দাও।

CoreTiny
CREATE TABLE business_units (
  bu_id      TINYINT UNSIGNED NOT NULL AUTO_INCREMENT,
  code       CHAR(6)          NOT NULL,   -- 'FOOD','BVRG','PLAS','PACK','AGRI'
  name       VARCHAR(80)      NOT NULL,   -- 'PRAN Food Division'
  parent_bu  TINYINT UNSIGNED NULL,       -- Sub-division হলে parent_bu_id
  is_active  TINYINT(1)       NOT NULL DEFAULT 1,

CREATE TABLE order_items (
  item_id       BIGINT UNSIGNED  NOT NULL AUTO_INCREMENT,
  order_id      BIGINT UNSIGNED  NOT NULL,
  order_date    DATE             NOT NULL,   -- Denormalized! Partition pruning-এর জন্য
  product_id    INT UNSIGNED     NOT NULL,
  batch_no      VARCHAR(20)      NULL,
  qty_ordered   DECIMAL(10,3)    NOT NULL,
  qty_delivered DECIMAL(10,3)    NOT NULL DEFAULT 0.000,
  unit_price    DECIMAL(10,2)    NOT NULL,
  discount_pct  DECIMAL(5,2)     NOT NULL DEFAULT 0.00,
  line_total    DECIMAL(12,2)    NOT NULL,   -- Denormalized: qty×price-discount

  PRIMARY KEY (item_id, order_date),
  KEY         idx_order_id      (order_id),
  KEY         idx_product_date  (product_id, order_date),
  FOREIGN KEY (product_id) REFERENCES products(product_id)

পার্ট ০৫

মিলিয়ন রেকর্ড অপটিমাইজেশন সিক্রেট

এই কৌশলগুলো না জানলে তোমার ডেটাবেস ১ কোটি রেকর্ডে ক্রল করবে। PRAN-এর মতো কোম্পানি এগুলো ব্যবহার করে।

🗂️
Partitioning
৩০ কোটি order-কে মাসে ভাগ করো। এপ্রিলের query শুধু এপ্রিল partition scan করে — ৯৬% data skip করে। EXPLAIN PARTITIONS দিয়ে verify করো।
🧠
Buffer Pool Tuning
innodb_buffer_pool_size = RAM × 70%। ৬৪ GB RAM সার্ভারে ৪৫ GB Buffer Pool। Hot data সবসময় RAM-এ থাকে — disk I/O প্রায় শূন্য।
📊
Covering Index
Report query-তে SELECT শুধু indexed column নাও। MySQL data page-ই যাবে না, index থেকেই answer দেবে। ৫-১০x দ্রুত হয়।
🔄
Read Replica
Write → Master DB। Report/Analytics → Slave DB। PRAN-এ report query master-কে block করে না। Binary Log-এ replication হয় real-time।
⚡
Denormalization (যেখানে দরকার)
order_items-এ order_date রাখা denormalization। কিন্তু এটা partition pruning-কে সম্ভব করে। ৩NF ভাঙা পাপ না — performance জরুরি হলে করো।
🗃️
Slow Query Log
slow_query_log=ON, long_query_time=1 — ১ সেকেন্ডের বেশি query log হয়। সেগুলো EXPLAIN দিয়ে fix করো। Production এর প্রথম কাজ এটাই।
গুরুত্বপূর্ণ Query — PRAN-RFL Daily Reports 
-- ১. দৈনিক sales summary per BU (Partition Pruning কাজ করবে)
SELECT
  bu_id,
  COUNT(*) AS total_orders,
  SUM(total_amount) AS total_sales,
  SUM(paid_amount) AS collected,
  SUM(total_amount - paid_amount) AS outstanding
FROM sales_orders
WHERE order_date = CURDATE()   -- partition pruning: শুধু আজকের partition
GROUP BY bu_id;

-- ২. Top 10 products this month (Covering Index ব্যবহার)
SELECT
  oi.product_id,
  SUM(oi.qty_delivered) AS qty_sold,
  SUM(oi.line_total) AS revenue
FROM order_items oi
WHERE oi.order_date BETWEEN '2026-04-01' AND '2026-04-30'
GROUP BY oi.product_id
ORDER BY revenue DESC
LIMIT 10;

-- ৩. Customer outstanding balance
SELECT
  c.name,
  SUM(so.total_amount - so.paid_amount) AS balance
FROM customers c
JOIN sales_orders so ON c.customer_id = so.customer_id
WHERE so.payment_status != 'PAID'
  AND so.order_date >= '2025-01-01'  -- partition range বলো
GROUP BY c.customer_id
HAVING balance > 0
ORDER BY balance DESC;

-- ৪. Expiry alert — ৩০ দিনের মধ্যে expire
SELECT
  p.name_bn, p.sku,
  i.batch_no, i.exp_date,
  i.qty_available,  -- Virtual column!
  w.name AS warehouse
FROM inventory i
JOIN products   p ON i.product_id   = p.product_id
JOIN warehouses w ON i.warehouse_id = w.warehouse_id
WHERE i.exp_date BETWEEN CURDATE() AND DATE_ADD(CURDATE(), INTERVAL 30 DAY)
  AND i.qty_available > 0
ORDER BY i.exp_date ASC;

পার্ট ০৬ — চূড়ান্ত জ্ঞান

প্রোর মতো ভাবার ১০টি সোনার নিয়ম

  • ১
    সবচেয়ে ছোট সম্ভব datatype ব্যবহার করো
    PRAN-এর bu_id TINYINT। district_id SMALLINT। order_id BIGINT। ভুল datatype = GB waste + ধীর index।
  • ২
    Primary Key সবসময় UNSIGNED AUTO_INCREMENT রাখো
    UUID ব্যবহার করো না PK হিসেবে — B+Tree fragmentation হয়, INSERT ধীর হয়। Sequential INT/BIGINT সবচেয়ে দ্রুত।
  • ৩
    টাকার জন্য সবসময় DECIMAL, কখনো FLOAT নয়
    FLOAT-এ ৳১০০.৫০ → ৳১০০.৪৯৯৯৯৯৭ হতে পারে। লক্ষ লক্ষ transaction-এ error জমে বিশাল হিসাব গড়বড় হয়।
  • ৪
    WHERE clause-এ যে column আছে সেখানে INDEX দাও
    কিন্তু বেশি index = INSERT/UPDATE ধীর। ৫-৭টার বেশি index না দেওয়াই ভালো। EXPLAIN দিয়ে verify করো।
  • ৫
    মিলিয়ন রেকর্ড টেবিলে PARTITION BY RANGE (date)
    এটাই সবচেয়ে বড় performance boost। ১২ মাসে ভাগ করলে প্রতি query ৯০%+ data skip করতে পারে।
  • ৬
    innodb_buffer_pool_size = RAM-এর ৭০%
    এটা না করলে MySQL বারবার disk থেকে read করবে। RAM থেকে data পড়া disk থেকে ১০০০x দ্রুত।
  • ৭
    SELECT * কখনো production-এ নয়
    শুধু দরকারী column select করো। Covering index কাজ করবে। Network bandwidth কমবে। Memory কম লাগবে।
  • ৮
    created_at/updated_at সবসময় TIMESTAMP দাও
    debug, audit, soft delete — সব কাজে লাগে। DATETIME-এর চেয়ে ৪ বাইট ছোট। ON UPDATE CURRENT_TIMESTAMP দাও updated_at-এ।
  • ৯
    Slow Query Log চালু রাখো সবসময়
    long_query_time=1 সেট করলে ১ সেকেন্ডের বেশি query লগ হয়। প্রতি সপ্তাহে log দেখো, EXPLAIN দিয়ে fix করো।
  • ১০
    EXPLAIN আগে, optimize পরে
    অনুমান করে index দিও না। EXPLAIN SELECT... চালাও। rows, type, key দেখো। type=ALL মানে full scan — সমস্যা। type=ref মানে index চলছে — ভালো।
